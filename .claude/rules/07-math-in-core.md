# 07 — Математика и модели — ТОЛЬКО в core/ (не в demo, не в JS)

> Правило без `paths:` — **грузится всегда**. Критическое (прецедент 2026-07-19).

## Правило

Любая **математика и модели** (камера, проекция, преобразование координат, сцена,
геометрия, DSP-формулы, кинематика) пишутся **только в `core/`** как ООП
(Value Object / Strategy, SOLID / GRASP / GoF) + тесты `common.runner.TestRunner`.

`demo/` и визуализация (в т.ч. **JS внутри web-страниц**) — **тонкий слой показа**
поверх `core`. Без собственной математики. Без дублирования формул в двух местах.

## Почему (прецедент 2026-07-19, ex4 web-панель)

3D-проекция и 2D-поле были размазаны в demo JS: знаки осей заданы в ДВУХ местах
(`fx/fy` поля и матрица `p3` в 3D). Источник конвенции не один → рассинхрон →
**зеркальность** → часы правок СЛЕДСТВИЯ (инверсии, clamp вращения) вместо ПРИЧИНЫ.

Причина одна: **не было единой модели камеры** (одного источника знаков осей).

## Как правильно (СРАЗУ, а не после костылей)

1. **Единая модель сразу** — один источник истины. Не костылять локальными
   инверсиями/подгонками «по месту» (`-u*s`, `NX/2-kx`, clamp) — это лечение симптома.
2. **Camera/Projection = Value Object** в `core/graphics/` — конвенция знаков осей
   задана ОДИН раз (`+kx → влево, +ky → вверх, дальность → вглубь`).
3. Все виды (2D и 3D) — проекции **ОДНОЙ** камеры ⇒ согласованность **by construction**,
   зеркальность невозможна в принципе.
4. **Тест доказывает свойство** (знаки совпадают, нет зеркальности, r=0 ближе), а не
   подгоняется скриншотами на глаз.

## Сигнал «стоп, ты костыляешь»

Если правишь одно место и ломается другое (лево↔право, знак, зеркало) — **остановись**:
это дублированный источник истины. Вынеси в единый VO в `core/`, не подкручивай следствие.

→ Реализация: [`core/graphics/camera.py`](../../core/graphics/camera.py) (`Projection`),
тест [`tests/test_camera.py`](../../tests/test_camera.py).
