# ECCM 3FFT — Архитектура (C4 model)

> Нотация — **C4** (Simon Brown): 4 уровня масштаба.
> **C1 System Context** → **C2 Container** → **C3 Component** → **C4 Code**.
> Стержень: одно ядро (объёмный токенизатор → гейт → причинностный арбитр → опрос FM-m),
> сменные интерфейсы под тип сигнала; токен на объёме (`16×16×L`) — универсальный интерфейс.

## C1 — System Context

Кто взаимодействует с системой и через что.

```
        ┌───────────────────────────┐
        │  Электромагнитная          │  цель + DRFM-самоприкрытие + RFI
        │  обстановка                │
        └─────────────┬─────────────┘
             отражения │ ▲ излучение (ЛЧМ/AM/FM-m)
                       ▼ │
   ┌───────────────────────────┐        ┌───────────────────────────┐
   │ Приёмный тракт             │  IQ/   │ Формирователь зондов       │
   │ АФАР + АЦП + дечирп(ПЛИС)  │──куб──► │ ЦАП (ЛЧМ/AM/FM-m, код)     │
   └─────────────┬─────────────┘        └─────────────▲─────────────┘
                 │ куб 16×16×L                          │ команда зонда/кода
                 ▼                                      │
        ┌════════════════════════════════════════════════════════┐
        ║  ★ СИСТЕМА ECCM-РАСПОЗНАВАНИЯ 3FFT  (в фокусе)          ║
        ║    распознаёт истинную цель, выдаёт целеуказание        ║
        └════════════════════════┬═══════════════════════════════┘
                                 │ целеуказание (углы, дальность)
                                 ▼
                    ┌───────────────────────────┐
                    │ Оператор / боевая система  │
                    └───────────────────────────┘
        (инфраструктура: core/DrvGPU — GPU-платформа, вне фокуса)
```

| Актор / внешняя система | Отношение к ECCM |
|---|---|
| Электромагнитная обстановка | источник отражений и помех (DRFM/RFI) |
| Приёмный тракт (АФАР+АЦП+дечирп ПЛИС) | **поставляет** куб `16×16×L` / IQ |
| Формирователь зондов (ЦАП) | **получает** от ECCM команду зонда/кода; излучает |
| Оператор / боевая система | **получает** целеуказание |
| core/DrvGPU | GPU-платформа (IBackend/GpuContext) |

## C2 — Containers

Крупные исполнимые блоки внутри системы и связи между ними (data ─►, control ⇢).

```
        куб 16×16×L
   ┌────────────────────┐
   │ [C2.1] ФРОНТЕНД     │  ЛЧМ-интерфейс | AM-интерфейс   [GPU]
   └─────────┬──────────┘
             │ ─► куб
   ┌─────────▼──────────┐
   │ [C2.2] ЯДРО-        │  токенизатор (инвариант к фронтенду)  [GPU/LDS]
   │        ТОКЕНИЗАТОР  │
   └─────────┬──────────┘
             │ ─► матрица токенов
   ┌─────────▼──────────┐   уточн.(угол,дальн.)   ┌────────────────────┐
   │ [C2.3] РАСПОЗНАВАНИЕ│◄───────────────────────│ [C2.4] ОПРОС FM-m   │
   │ гейт+арбитр+L3      │── ⇢ пучок/код ─────────►│ коррелятор+кодоген  │
   └─────────┬──────────┘   отклик (подтвержд.)   └────────────────────┘
             │ ─► кандидаты (метка цель/ложь)
   ┌─────────▼──────────┐
   │ [C2.5] ЦЕЛЕУКАЗАНИЕ │  beam-former + когнитивная петля  [CPU/control]
   └─────────┬──────────┘
             │ ⇢ следующий такт (новый код/N) ──► [C2.1]
   [C2.6] ИНФРА core/DrvGPU ── IBackend/GpuContext ──► всем контейнерам
```

| # | Контейнер | Технология | Ключевые связи |
|---|---|---|---|
| C2.1 | Фронтенд (ЛЧМ/AM) | GPU (rocFFT/WMMA) | приёмный тракт →; ─► **куб** в C2.2 |
| C2.2 | Ядро-токенизатор | GPU / LDS | куб →; ─► **матрица токенов** в C2.3 |
| C2.3 | Распознавание (гейт+арбитр+L3) | GPU + CPU | токены →; ⇢ **пучок/код** в C2.4; ◄ отклик из C2.4; ─► кандидаты в C2.5 |
| C2.4 | Опрос FM-m (коррелятор+кодоген) | GPU (hipFFT) | ◄ команда из C2.5; ─► отклик в C2.3 (обратная связь) |
| C2.5 | Целеуказание + петля | CPU-control | ◄ кандидаты; ⇢ команда зонда; ⇢ **новый такт** в C2.1 |
| C2.6 | Инфра DrvGPU | core lib | IBackend/GpuContext, память, StreamPool, профилирование — всем |

## C3 — Components

Зум в два ключевых контейнера.

### C3.a — Ядро-токенизатор (C2.2)

```
куб ─► AngularFFT(WMMA/MFMA) ─► MagnitudeShift(|·|²+fftshift)
        ─► FeatureExtractor(PR,Hoyer,лепестки) ─► OS-CFAR(порог)
        ─► PeakFinder(≤5 пиков[+kz]) ─► Tokenizer ─► матрица токенов
        (всё в LDS, без выхода в VRAM)
```

| Компонент | Назначение | Вход → выход |
|---|---|---|
| `AngularFFT` | угловой БПФ 16×16 на тензорных блоках | куб → карта ℂ |
| `MagnitudeShift` | `|·|²` + fftshift | ℂ → энергия ℝ |
| `FeatureExtractor` | страддл-инвариантные признаки | карта → вектор |
| `OS_CFAR` | порог выброса (2D/3D) | карта → метка |
| `PeakFinder` | до 5 пиков по углу (+kz) | карта → пики |
| `Tokenizer` | сборка токена | пики+признаки → токен |

### C3.b — Опрос FM-m (C2.4)

```
AgileCodeGen ─► ForwardFFT(ref) ┐
вход ─► ForwardFFT(inp) ─► ConjMultiply ─► InverseFFT ─► IfftTokenizer(3–5)
              BatchManager ⇢ нарезка,  StreamPool ⇢ 2 потока
```

| Компонент | Назначение | Связи |
|---|---|---|
| `AgileCodeGen` | M-код 2⁸…2²⁰ по такту | → эталон ref |
| `ForwardFFT/ConjMultiply/InverseFFT` | корреляция S×K через БПФ | ref+вход → отклик |
| `IfftTokenizer` | 3–5 максимумов → токены | отклик → пики |
| `BatchManager` / `StreamPool` | батчи по памяти / 2 потока | инфра C2.6 |

## C4 — Code (Ref03)

Компоненты ложатся на 6-слойную модель **Ref03** репозитория `core`:

| Ref03 | Класс/ядро ECCM |
|---|---|
| 1 `GpuContext` | общий контекст пайплайна (backend, stream, BufferSet<8>) |
| 2 `IGpuOperation` | контракт каждой операции (KernelA, KernelB, corr-ops) |
| 3 `GpuKernelOp` | база HIP-ядер: `RangeFftOp`, `AngularTokenizeOp`, `CorrelateOp` |
| 4 `BufferSet<N>` | буферы куба / карты / токенов / откликов |
| 5 concrete Ops | `KernelA_RangeFFT`, `KernelB_Tokenizer`, `FmCorrelator`, `L3_TokenLSTM` |
| 6 Facade+Strategy | `EccmPipeline` (radar/strategies) + стратегия фронтенда (ЛЧМ/AM) |

## Сквозные связи и инварианты

- **фронтенд → ядро только через куб** `16×16×L` — сменить фронтенд, не трогая ядро;
- **ядро → распознавание только через матрицу токенов** — универсальный интерфейс;
- **гейт ≠ арбитр**: MLP — приоритет, метку ставит физика (τ≥0 / код);
- **обратная связь C2.4 → C2.3** (свежесть кода доступна после опроса; в устройстве — 4.5→4.4);
- **когнитивная петля C2.5 → C2.1** (новый такт, новый код/длина/N); «летит» — межтактовым трекингом;
- инфра `core/DrvGPU` — под всеми контейнерами (IBackend/GpuContext), не в фокусе алгоритма.
