## 1. Что это за проект

Разработка **способа и устройства** каскадного распознавания цели на фоне
**самоприкрывающих помех** для моностатического UHF-радара (0.5 ГГц) с АФАР.
Цель работы прямо сейчас — **материал под статью и патент** и показать алгоритм
коллегам. Генератор сцен и обучение сетей — потом.

**Угроза:** DRFM/ISRJ (гребёнка ложных целей, ярче истинной), SMSP (размыв
спектра), заградка, сторонний непрерывный источник (RFI, «радиолюбитель»).

**Прототип (уровень техники):** RU2549375C1 — не тянет самоприкрытие (нужен
отдельный ретранслятор, геометрия Δt, нет структурного различителя).

**Приоритеты (именно в этом порядке):**
надёжность распознавания → скорость → красота решения.

**Железо/стек:** AMD RX 9070 (RDNA4/gfx1201, 16 ГБ), ROCm 7.2, HIP, rocFFT/hipFFT,
C++. Есть рабочий FM-m коррелятор в репозитории `AlexLan73/GPUWorkLib`
(`modules/fm_correlator`).

## 2. Полная цепочка обработки (согласована с пользователем)

```
АФАР → АЦП → гетеродин-дечирп на ПЛИС (stretch) → сырые IQ
   → децимация → окно FFT
   → БПФ (по всем антеннам батчем на GPU)
   → быстрый триаж по SNR (моментный метод M2M4)
       ├─ чистый сигнал (хороший SNR):
       │     FFT → макс. гармоника → 3-точечная парабола (EA3) → частота → дальность
       │     → фазирование ФАР для наведения
       └─ грязный (шум / много целей) → КАСКАД:
             L1  грубое обнаружение на угловой карте
             L2  физическое ядро (передний край + M-код + задержка петли)  ← сердце патента
             L3  глубокий добор
             → целеуказание пучка FM+M
```

**Ключевые физические факты (не потерять):**
- После дечирпа **частота биений = дальность**: `f_b = μ·2R/c`, `μ = ΔF/T_c`.
  Одна цель → один пик; гребёнка ложных → ряд пиков с шагом Δf.
- **Разрешение по дальности:** `Δr = c/(2·ΔF)` (в знаменателе **девиация ΔF**,
  НЕ база `B = T_c·ΔF`. База — это помехозащищённость, не разрешение).
- **3-точечная парабола (EA3)** оптимальна именно для острого дечирп-пика
  (проверено в отдельном SNR-исследовании): `δ = ½·(y₋₁−y₊₁)/(y₋₁−2y₀+y₊₁)`.
- **SNR моментным методом:** `κ = M4/M2² − 2`, `SNR = (1−κ)/κ`
  (у ЛЧМ постоянная огибающая κ→0, у шума κ→1).

## 3. Геометрия каскада и ДВА FFT (тут была главная путаница — разрешена)

Вход в каскад: тензор **апертура × апертура × дальность** (в текущей модели
`16×16×10000`).

> **Два разных FFT в разных местах конвейера, НЕ один 3D-FFT:**
> 1. **Дальностный FFT** — глобальный, ОДИН раз, по всей оси быстрого времени
>    (rocFFT). Превращает биения в дальность. После него Z = дальность.
> 2. **Угловой FFT 16×16** — поячеечно, на каждом бине дальности (WMMA/тензор).

- Нарезка «окно 16, шаг 8, overlap 50%» — **не нужна** (артефакт старого
  понимания): после дечирпа цель — тон на всю запись, STFT дальность не
  локализует.
- Магнитуду `|·|²` берём **только после углового FFT** (он когерентно
  складывает по апертуре — ему нужна фаза).
- Угол при `d = λ/2`: `sinθ = k/8` (k = −8…7), покрытие полусферы, разрешение
  по sinθ = 1/8 (≈7° у визирования).
- Метрика **параметрическая** (меняется на ходу, алгоритм постоянен).

## 4. L1 — грубое обнаружение (глава уже написана)

- Классифицируем не по сырой карте 16×16, а по **вектору инженерных признаков**.
- Признаки (проверены на числах): **PR** `=(Σp)²/Σp²`, **Hoyer**, **доля
  главного лепестка (3×3)**, **интегральное отношение лепестков** (это два
  страддл-устойчивых признака, заменили «сырой 2/1 пик», который ломался на
  страддле), max/mean, энергия. `p_i = |A_i|²`, n=256.
- **Устойчивые якоря:** PR, Hoyer, доля лепестка, интегр. отношение лепестков.
  Проверено: цель на сетке и в страддле по ним неотличимы, а от заградки
  отделены чисто. Есть 3 графика (feat_scene, straddle, intfeat).
- **Токен со структурой:** массив до 5 пиков `(kx,ky,яркость,кромка)` +
  признаки + метка. Две ортогональные оси: **массив пиков — по углу**,
  **сборка токенов — по дальности**.
- **Триаж MLP** 6→16→3: шум / собранный / размазанный. Это **ГЕЙТ, не арбитр** —
  метку «цель» ставит физика L2 (DRFM даёт «собранный», для угловой сети
  неотличимый от цели).
- **Проход 2** (сборка вдоль дальности): передний край (ближний ≠ яркий) +
  период гребёнки (автокорреляция цепочки токенов).

## 5. L2 — физическое ядро (СЕРДЦЕ ПАТЕНТА)

- **Правило переднего края:** истинная цель — ближний пик группы; DRFM причинно
  только задерживает → ложные всегда ДАЛЬШЕ. Ось односторонняя, **τ ≥ 0**.
- **Агильный M-код:** КАЖДЫЙ ТАКТ новый код и **новая длина**, приводимая
  **нулями к записи 2^n** (zero-pad до размера FFT). `L_t ≤ N`.
- **НЕ большой банк кодов** (RAM не хватит), а **формирование по команде**: от
  решения о коде/длине до прихода данных успеваем построить сопряжённые опоры и
  настроить планы FFT. Это соображение безопасности.
- **Короткая скользящая история** глубиной `K` тактов;
  **`K` = максимально измеримая задержка петли** (проектный параметр).
- **Различение текущий/исторический код:** цель откликается на ТЕКУЩЕМ коде
  (слот 0); репитер несёт УСТАРЕВШИЙ код (слот d) — не успел за сменой.
  Подавление рассогласования ≈ `10·log₁₀(L)` дБ.
- **Оценка задержки петли** = номер кода/слота d, где сработал репитер.
  Это ГЛАВНЫЙ отличительный признак заявки.
- **Причинность = неподделываемый признак:** враг не ответит кодом, которого
  ещё не слышал.
- Алгоритм описан **на один такт**; дальше цикл повторяется с новым кодом.

## 6. FM-m коррелятор (реализация L2, код готов — описание написано)

- Корреляция через БПФ: `corr = IFFT( conj(FFT(ref)) · FFT(inp) )`, O(N log N).
- **Два HIP-потока:** эталон (stream0) и вход (stream1) параллельно; эталон
  готовится ОДИН раз и переиспользуется.
- **4 ядра:** циклические сдвиги, сопряжённое умножение, извлечение магнитуд,
  генератор тест-входов. Микрооптимизации: aligned float2, побитовый модуль,
  FMUL вместо FDIV, эрмитова симметрия (half_N = N/2+1).
- **Батчинг по памяти (BatchManager):** влезает всё → один проход; не влезает →
  режет на батчи по ~70% свободной памяти, эталон не трогается, пересоздаются
  только буферы входа, данные подаются прямым указателем без копий.
- **ДВА РЕЖИМА ФИНАЛА (важно):**
  - **банк задержек** (K сдвигов ОДНОГО кода) → корреляции = сдвиги одной →
    коллапс `S×K → S` IFFT. **Только для теста/калибровки.**
  - **банк истории кодов** (разные коды по тактам) → коллапс НЕПРИМЕНИМ →
    честные `S×K` IFFT. **Это боевой режим.** Текущий код пользователя уже
    делает `S×K` — и это ПРАВИЛЬНО, не «недооптимизировано».

> Был эпизод: пользователь переживал, что я назвал `S×K→S` «ускорением», а
> другой Claude (Кодо) сказал, что его код правильный. ОБА правы: код
> правильный, а коллапс — необязательная оптимизация, применимая ТОЛЬКО когда
> опоры суть сдвиги одного кода. В бою коды разные → коллапс не работает.
> Не запутай его снова: его алгоритм верен.
