# FM-m коррелятор: алгоритм и архитектура

## Корреляционное ядро распознавания цели по фазовому M-коду на GPU

**Модуль:** `modules/fm_correlator` · **Backend:** ROCm (hipFFT/rocFFT + HIP)
**Назначение документа:** материал под патент (способ/устройство) и доклад коллегам

---

## 1. Идея в двух строчках

Радар излучает сигнал, фазово-манипулированный **M-последовательностью** —
псевдослучайным кодом с почти идеальной автокорреляцией. Приёмник **коррелирует**
принятое с эталонным кодом: там, где корреляция даёт острый пик, — там цель, а
**положение пика по оси задержки = дальность**. Всё это считается на GPU в
частотной области, пакетами по многим лучам сразу.

Модуль отвечает на вопрос: *«есть ли в принятом сигнале наш код, и с какой
задержкой»* — и делает это для `S` входных лучей против `K` гипотез задержки
одновременно.

---

## 2. Почему M-последовательность

M-последовательность (генерируется регистром сдвига с линейной обратной связью,
LFSR, по примитивному полиному) — это код из `±1` с двумя ценными свойствами:

- **острая автокорреляция**: корреляция кода с самим собой — почти дельта-функция
  (высокий центральный пик, боковые лепестки уровня `~1/√N`). Значит совпадение по
  задержке видно как чёткий одиночный пик, а не размытое пятно;
- **псевдошумовой спектр**: код выглядит как шум, что даёт помехозащищённость и
  скрытность.

Генерация — на CPU (код нужен один раз), регистр Галуа: один сдвиг и один XOR с
полиномом на бит, бит `→ ±1.0`. Результат уходит на GPU и там переиспользуется.

---

## 3. Математика: корреляция через БПФ

Прямая корреляция двух сигналов длины `N` стоит `O(N²)`. По теореме о свёртке её
заменяют тремя быстрыми преобразованиями:

```
corr(ref, inp) = IFFT( conj(FFT(ref)) · FFT(inp) )
```

Стоимость падает до `O(N log N)`. Пошагово:

1. `FFT(ref)` — спектр эталона (готовится заранее, один раз);
2. `FFT(inp)` — спектр принятого сигнала;
3. поэлементное `conj(FFT(ref)) · FFT(inp)` — сопряжение эталона превращает
   свёртку в корреляцию;
4. `IFFT(...)` — обратно во время: получаем корреляционную функцию;
5. `|·| / N` — магнитуда и нормировка; ищем пики.

Пик в точке `j` означает: принятый сигнал содержит эталонный код, задержанный на
`j` отсчётов. Задержка `j` пересчитывается в дальность.

---

## 4. Банк из K сдвигов — и почему это сердце патента

Эталон готовится не в одном экземпляре, а как **банк из `K` версий** — `K`
циклических сдвигов кода. Каждый сдвиг — это **отдельная гипотеза задержки**.
Коррелятор проверяет все `K` гипотез за один проход: для входного луча `s` и
сдвига `k` получаем свой корреляционный отклик, а положение пика по `(k, j)`
локализует цель по дальности.

Здесь ключевое обобщение, важное для патента. Тот же самый банк из `K` опорных
спектров — это универсальный движок, и в него можно загрузить **не циклические
сдвиги одного кода, а `K` разных исторических кодов** (по одному на прошлые
импульсы агильной кодовой последовательности). Тогда:

- пик на **текущем** коде (слот 0) → **истинная цель**;
- пик на **устаревшем** коде (слот `k>0`) → **ретранслятор (DRFM)**, который
  повторяет наш прошлый код с задержкой петли;
- **номер слота `k`, на котором сработал пик, = оценка задержки петли** помехопостановщика.

То есть реализованный коррелятор — это готовый **аппарат банка гипотез**: сегодня
он работает как банк задержек (K сдвигов одного кода), а физическое ядро патента
(текущий-vs-исторический код + оценка петли) получается **той же машиной**, если
в `K` слотов подать историю кодов вместо сдвигов. Архитектура для этого уже готова:
подготовка `K` опорных спектров, умножение `S×K`, извлечение пиков — не меняются.

---

## 5. Архитектура GPU: два потока и переиспользование эталона

Обработка построена на **двух независимых потоках** ради настоящего параллелизма:

```
поток 0 (эталон):   H2D(ref) → циклические сдвиги → FFT(K сдвигов)
поток 1 (вход):     H2D(inp) → FFT(S сигналов)         ← идёт одновременно
                    ── синхронизация обоих потоков ──
поток 0 (финал):    conj·умножение → IFFT(S×K) → магнитуды → D2H
```

Два выигрыша:

- **эталон готовится один раз** и живёт на GPU: его `K` спектров переиспользуются
  во всех последующих вызовах без повторной загрузки;
- пока поток 1 грузит и преобразует входные лучи, поток 0 уже сделал свою часть —
  латентность прячется за перекрытием.

Планы БПФ создаются один раз при установке параметров (создание плана дорогое) и
привязываются к своим потокам, чтобы не сваливаться на нулевой поток и не рушить
параллелизм.

---

## 6. Четыре вычислительных ядра

Весь счёт — четыре компактных HIP-ядра, скомпилированных на целевую архитектуру
прямо на устройстве (оптимальный код под конкретный GPU, с дисковым кешем бинаря):

1. **Циклические сдвиги** — из вещественного эталона строит `K×N` комплексный
   банк: `out[k][i] = ref[(i+k) mod N]`. Сдвиг по модулю `N` — побитовое `И`
   (N — степень двойки), без деления.
2. **Сопряжённое умножение** — `conj(ref_fft[k]) · inp_fft[s]` за один проход по
   всем парам `(s, k)`. Сопряжение — инлайн (смена знака мнимой части), без вызова
   библиотечных функций.
3. **Извлечение магнитуд** — `|corr[j]| / N` для первых `n_kg` точек (дальние
   задержки не нужны — экономим память и время). Модуль — побитовым сбросом знака
   (одна инструкция, без ветвления); деление на `N` заменено умножением на `1/N`.
4. **Генератор тест-входов** — служебное: строит `inp = circshift(ref, s·шаг)`
   прямо на GPU для самопроверки без загрузки данных с хоста.

Микрооптимизации в тон задаче: выровненный `float2` (одна 64-битная транзакция
вместо двух 32-битных); использование эрмитовой симметрии вещественного БПФ
(`N/2+1` точек вместо `N` — экономия ~50% памяти на спектрах входа); нормировка
умножением, а не делением.

---

## 7. Батчинг по размеру: считать всё сразу, если влезает

Это, пожалуй, самая изящная часть инженерии. Число входных лучей `S` бывает
большим, и весь массив может не поместиться в память GPU. Система решает это
**сама**, без ручного вмешательства:

- **менеджер батчей** оценивает, сколько лучей влезает в `70%` свободной памяти
  GPU, заранее вычтя постоянно занятый банк эталона (`K` спектров всегда на месте);
- **влезает всё — считает всё за один проход** (батч = весь массив);
- **не влезает — режет на батчи** оптимального размера и обрабатывает их по
  очереди, накапливая результат;
- **эталон при этом не трогается** — он подготовлен один раз и остаётся на GPU;
  между батчами пересоздаются только буферы под входные лучи, и то лишь когда
  меняется размер батча (обычно один-два раза: все батчи одного размера, кроме
  последнего);
- данные подаются **прямым указателем** в нужное смещение массива — без лишних
  промежуточных копий на хосте.

Итог: один и тот же вызов корректно отрабатывает и пять лучей, и тысячи —
прозрачно для пользователя, с максимальным использованием доступной памяти и без
переполнения. Масштаб подстраивается под железо, а не наоборот.

---

## 8. Что на выходе

Массив пиков `peaks[S × K × n_kg]`: для каждого луча `s`, каждой гипотезы `k` и
каждой задержки `j` — магнитуда корреляции. Дальше по нему:

- **есть ли цель** — по превышению пика над фоном;
- **дальность** — по положению пика `j`;
- **истинная vs ложная** — по тому, на текущем или на историческом коде (слоте `k`)
  сработал пик;
- **задержка петли DRFM** — по номеру сработавшего слота.

Это ровно те величины, которые физическое ядро каскада (уровень L2) использует для
целеуказания активного FM+M зондирования.

## 9. Заметки для патента

- **Прототипы (уровень техники).** В документации модуля указаны две работы по
  GPU-ускоренной кросс-корреляции для пассивного бистатического радара
  (Remote Sensing, MDPI 15(22):5421; J. Parallel Distrib. Comput., 2025). Они
  описывают саму GPU-корреляцию — её и следует брать как ближайший уровень техники.
- **Что заявляем как новое** (не сама GPU-корреляция, она известна): агильный
  пер-импульсный код + **банк истории кодов** (текущий против прошлых) +
  **оценка задержки петли** по сработавшему слоту + разделение истинной цели и
  ретранслятора по «переднему краю» и по коду. Коррелятор — реализующий это движок.
- **Сверить перед подачей.** В коде и документации расходятся параметры LFSR
  (документация: степень 31, полином `0xB8000000`; конфигурация по умолчанию:
  степень 32, полином `0x00400007`, `fft_size = 32768`). Для воспроизводимости
  формулы в заявке зафиксировать канонический полином и степень.
- **Заявленная оптимизация (в работе).** Поскольку `K` опорных версий — это
  циклические сдвиги одного кода, все `K` корреляций суть циклические сдвиги одной
  базовой: можно считать **одно** обратное БПФ на луч и брать `K` оконных чтений
  со сдвигом вместо `S×K` обратных БПФ. Ожидаемое резкое сокращение времени финала.

---

## 10. Одним абзацем для доклада

Коррелятор берёт принятые лучи и известный псевдошумовой M-код, за счёт свёртки
через БПФ ищет совпадение по всем гипотезам задержки сразу, и по острому
корреляционному пику говорит «цель здесь, дальность такая». Он готовит эталон один
раз, считает в двух параллельных потоках, использует четыре плотных GPU-ядра и сам
подстраивает размер пакета под память видеокарты — считает всё разом, если влезает,
и по частям, если нет. А главное — тот же банк гипотез, что сегодня перебирает
задержки, завтра перебирает историю кодов, и это превращает быстрый коррелятор в
физический различитель истинной цели и ретранслятора — сердце всего метода.
