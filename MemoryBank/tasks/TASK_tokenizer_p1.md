## 0. Глубокое ревью: ошибки исходного мини-плана (ИСПРАВЛЕНЫ)

Первый набросок плана содержал 6 расхождений с патентом — все учтены ниже:

| # | Ошибка в наброске | Что в патенте (истина) |
|---|-------------------|------------------------|
| **E1** | один `VolumeToken{k_x,k_y,k_z}` на весь куб (1 пик) | токен **на срез дальности** `r`, внутри — **массив до 5 угловых пиков**; пустые («шум») срезы НЕ пишутся → выход разрежённый (гл.4 §4.6–4.7) |
| **E2** | план обрывался на пикселе-признаках (1 проход) | **ДВА прохода**: проход 1 — признаки+метка на срезе; **проход 2 — сборка по дальности** (цель/гребёнка/заград через автокорр Δr), гл.4 §4.9 — этого не было |
| **E3** | цель/заград различались на шаге признаков | проход 1 даёт лишь **{шум/собранный/размазанный}**; **цель↔гребёнка** различает **только проход 2** (по дальности) |
| **E4** | LobeRatio: «обнулить 5×5, argmax/главный» | `LobeRatio = Σ_{3×3 вокруг 2-го лепестка} P / Σ_{3×3 вокруг главного} P` (2-й лепесток ищем ПОСЛЕ обнуления охранной зоны 5×5), гл.4 §4.5 |
| **E5** | «OS-CFAR заменяет −10 дБ» без уточнения | OS-CFAR — это **гл.4-бис** (объёмный порог train/guard-воксели). Плоский L=1 (гл.4) — поиск пиков + мягкий порог. OS-CFAR 3D = порог выброса в 3D |
| **E6** | триаж = RuleBased как основной | патент: **MLP 6→16→3**. Для torch-free прототипа берём **RuleBased по таблице §4.11** как замену (LSP: позже MLP), это осознанная девиация — задокументировать |

Дополнительно (не ошибка, но критично для реализации):
- **E7 · Угловой FFT уже в кубе.** `LfmToCube`/`AmToCube` уже дали beamspace →
  токенизатор берёт `SpectralCube` и НЕ гоняет угловой FFT повторно. `P = magnitude²`
  (`SpectralCube.magnitude` = `|A|`, не `|A|²`!). Двойного FFT быть не должно.

## 1. Что делаем

Новый модуль `core/models/tokenizer/` — **объёмный токенизатор** (Template Method, SPEC §3),
общий для обеих веток заполнения куба. Заменяет заглушку `SquareView.tokenize`
(фикс. −10 дБ + argmax) настоящим детектором из патента. Плоский случай (L=1) и
объёмный (L>1) — одно ядро, L=1 частный случай (гл.4-бис).

## 2. Контракт токена (кросс-язычно, под C++/msgpack, SPEC §6)

```
SliceToken = {              # ← на СРЕЗ дальности r (гл.4 §4.7), не на куб
    r: int,                 # индекс бина дальности (для L>1 — k_z окна)
    n_peaks: int,           # 0..5
    peaks: [ {kx,ky, amp, edge} ] * n_peaks,   # edge = кромка (нарастание/спад по r)
    f: {PR, Hoyer, MainFrac, LobeRatio, MaxMean, Energy},   # 6 признаков (§4.5)
    label: "noise"|"source"|"smeared",         # проход 1 (§4.8)
    score: float,
}
# Проход 2 (§4.9) добавляет над потоком SliceToken:
RangeVerdict = { kx, ky, kind: "target"|"comb"|"barrage", lead_r: int, period_dr: float|None }
```
> `SquareToken` (`square_view.py`) НЕ ломаем — оставляем как контрольный вид;
> `SliceToken`/`RangeVerdict` — новые VO. Пустые срезы не пишем (разрежённый выход).

## 3. Проход 1 — признаки + триаж на срезе (гл.4 §4.3–4.8)

Для каждого среза `P = |cube[:, :, r]|²` (M=nx·ny — фактическое число угловых ячеек куба
после zero-pad, для частного случая 16×16 → M=256; S1=Σpᵢ, S2=Σpᵢ²):

| Признак | Формула (патент §4.5, ПРОВЕРЕНО) | цель / заград / шум (§4.11) |
|---------|-----------------------------------|------------------------------|
| **PR** | `S1² / S2` | 3.6 / 15–23 / 129 |
| **Hoyer** | `(√M − S1/√S2)/(√M − 1)` (M — фактический, см. выше) | 0.94 / 0.81 / 0.31 |
| **MainFrac** | `Σ_{3×3 вокруг argmax} P / S1` | 0.98 / 0.40 / 0.07 |
| **LobeRatio** | `Σ_{3×3 у 2-го лепестка} / Σ_{3×3 главный}` (2-й — вне охр. зоны 5×5) | 0.002 / 0.25 / 1.03 |
| **MaxMean** | `max(P)/mean(P)` | 123 / 22–42 / 5.4 |
| **Energy** | `S1` | абсолют |

- **Порог выброса:** OS-CFAR (реюз идеи `core/models/anti_barrage/cfar.py` / `core/snr/estimator.py`).
  L=1 → 2D-CFAR по срезу; L>1 → **OS-CFAR 3D** (train/guard-воксели, гл.4-бис §4-бис.3:
  главный лепесток **3×3×3**, охранная зона **5×5×5**, PR/Hoyer по вокселям). ⚠️ Зоны 3×3/5×5 —
  граничный случай малых `N_pad` (nx или ny < 5): код усекает зону по краям куба (clip, не wrap).
- **Пики (до 5):** локальные максимумы над CFAR-порогом, координаты `(kx,ky)` + `amp` + `edge`
  (кромка из соседних бинов `r±1` — «почти бесплатно», §4.6). >5 пиков → сам по себе «размазано».
- **Триаж (гейт, не арбитр):** RuleBased по `f` → {noise/source/smeared} по разделяющим
  порогам таблицы §4.11 (главные якоря PR/Hoyer/MainFrac/LobeRatio — страддл-устойчивы, §4.11
  вторая таблица). `CubeClassifier`-совместимо (реюз ABC `classification/classifier.py`,
  LSP → позже `Cnn3DClassifier`/MLP). Порог записи токена **мягкий** (слабый ближний пик под
  яркой ложной не отсеиваем — передний край терять нельзя, §4.8).

## 4. Проход 2 — сборка по дальности (гл.4 §4.9)

Над потоком `SliceToken` с меткой `source` под одним `(kx,ky)`, разложенным по `r`:

| Картина токенов вдоль `r` | kind |
|---------------------------|------|
| одиночный токен | `target` (одиночная цель) |
| цепочка с равным шагом Δr (подтверждается **автокорреляцией** цепочки) | `comb` (гребёнка ретранслятора) |
| токены во всех `r` подряд | `barrage` (заград/сплошной) |

`lead_r = min r` группы → **передний край**, кандидат в истинную цель (метку цель/ложь
окончательно ставит арбитр гл.5 — вне P1). Работа по единицам токенов, не по кубу — дёшево.

## 5. Паттерны / SOLID (SPEC §3, §8)

- `VolumeTokenizer` — Template Method (`tokenize(cube) -> list[SliceToken]`), объём L≥1.
- `FeatureExtractor` — Pure Fabrication (6 признаков, чистая редукция).
- `CfarDetector` — Strategy (2D срез / 3D объём), реюз anti_barrage/snr.
- Триаж — `CubeClassifier`-совместимый RuleBased (LSP → MLP), реюз ABC.
- DI в Composition Root; куб читаем по ссылке, НЕ мутируем (чистота).

## 6. Проверка (🚫 pytest — TestRunner, правило 04)

   проверить попадание PR/Hoyer/MainFrac/LobeRatio/MaxMean в коридоры таблицы §4.11.
2. **Страддл-устойчивость** — цель на полбина: PR/Hoyer/MainFrac/LobeRatio почти не плывут
   (§4.11 вторая таблица); «сырой 2-й/1-й пик» ломается — подтвердить, что мы его НЕ используем.

   одиночная → `target`.
