## 🔬 Исследование: four-step / 3D-FFT на тензорных ядрах (2026-07-16, работа)

> Папка `tests/test-2D-FFT256/`. Цель Alex: доказать, можно ли строить угловую обработку
> целиком на матричных ядрах GPU. **Вывод: можно, доказано численно.**

**Разобрал эксперимент Alex `fourstep_v2.py`** (2D-FFT 256×256 из плиток 16×16, Bailey):
запустил, сверил математику, ревью → **склейка машинно-точна (8.9e-17)**, полное разрешение
(0.39° vs 7.18°), две стадии сами снимают неоднозначность (`k=k₁+16k₂`), NUFFT для скруглённой
апертуры не нужен. Ревью → `specs/fourstep_2dfft256_review_2026-07-16.md`.

**Проверил гипотезу 32×32 vs 16×16** (`fourstep32_compare.py`): обе машинно-точны; 32×32 лучше
на MI100 (−1 проход по VRAM для N=степень-32, вдвое тоньше угол, нативный MFMA). На RX 9070 —
16×16 (нативный WMMA). Точность four-step НЕ зависит от факторизации.

**Замерил точность fp16/bf16** (`fourstep32_fp16_dr.py`, эмуляция MFMA + fp32-acc):
**fp16 держит ДД −72 дБ** (≫ радарных −40…−60), bf16 −55 дБ (только детект). fp32-аккумуляция
ОБЯЗАТЕЛЬНА. → куб можно хранить в fp16 (−½ трафик VRAM).

**Расширил на 3D-FFT 256×256×D** (`fft3d_tensor.py`, D=8..128): сепарабельно → 3 GEMM;
малая ось Z — прямым DFT-GEMM (1 matmul, без twiddle). Машинно-точно; **fp16 по Z −72 дБ
НЕ зависит от D** (fp32-acc гасит √D); порог прямой↔four-step по Z ~D=64.

**Тайминги (roofline):** ЛЧМ-куб 256×256×6000 → **~7–11 мс на MI100** (fp16), ~20–40 мс RX 9070;
задача **memory-bound** (рычаг — fused-LDS + fp16, не арифметика). Один 3D-доплер-куб = 4–205 мкс
(практически бесплатно). MI100: L2 8 МБ спилит при D≥32; RX 9070: 64 МБ Infinity Cache → всегда
compute-bound.

**Документы:** сводный `00_MASTER_razbor_3FFT_tensor.md` (mermaid) + разборы для чайников/инженеров
+ `FFT3D_na_tenzorah.md`. Все .py запускаются на `.venv` (numpy 2.5).

**Хвост/дальше:** незакрытое — rocFFT-bench дальностной стадии на живой MI100 (заменить roofline-
прикидку замером). Дальностный FFT оставлен fp32 (длинный FFT в fp16 просадит боковики — не мерялось).
