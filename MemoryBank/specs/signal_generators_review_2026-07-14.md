## 🟡 Поправить в спеке

### R3. §4.6 «реюз Scene как SceneModeler» — неточно
Существующий `Scene.contribute` возвращает **куб** `(nx,ny,n_real)`, а SceneModeler должен отдавать
**параметры на элемент** (угол/задержка/амплитуда) для время-доменного генератора. Это **разные
контракты** → не drop-in реюз. Точнее: реюзаем `ArrayGrid.steering` (геометрия) и `*Spec` VO, но
**`SceneModeler` — новый** (или адаптер), а `Scene.contribute` (куб-уровень) не трогаем.

### R4. Разрыв единиц: `n_samples=8192` (время) ↔ `L=256` (глубина дальности)
В §5.1 baseline: 8192 отсчётов времени, но куб «16×16×256/16». Как 8192→256 бинов (дечирп→FFT→
дециация/окно)? Маппинг **время ↔ бин дальности** не задан. Куб отложен (§6), поэтому пока ок, но
**числа §5.1 иллюстративны** — пометить, что связку пиним на фазе куба.

### R5. Нет калибровки мощности/SNR (в дБ)
Датасет (industrial §8) откалиброван **в дБ, но по РАЗНЫМ шкалам** (не «весь в SNR −5..+25»):
SNR цели `−5..+25`, уровень VFD `−30..−60`, CW-помеха `A_CW −5..+15`, клаттер `SCR 0..+30`, шум `±10`.
В §4/§5 есть `AdditiveNoise`, но **дБ-калибровка на источник** не заложена явно. → добавить в
`WaveformSpec`/шум параметр `snr_db` (и реюз нашего `core/snr` для сверки). Без этого датасет не построить.
> ⚠️ **Мат-уточнение (важно на реализацию):** формула §4.4 `A=√(σ²·10^(SNR/10))` верна для
> **комплексного** сигнала (`P=|A|²`). Для вещественного `A·cos` (industrial §6.1) нужен множитель 2:
> `A=√(2·σ²·10^(SNR/10))`. Генерация — комплексный baseband (IQ) → без множителя, **но `σ²` считать
> как дисперсию комплексного шума** (I+Q), иначе амплитуда уедет в √2 раз.

### R6. Детерминизм датасета (seed) не прописан
79k примеров требуют воспроизводимости. Существующий `Synthesizer(seed=7)` — есть, но новые
`Waveform`/`AdditiveNoise`/`TactSequence` должны **явно принимать rng/seed**. → зафиксировать в §4:
все источники случайности — через переданный `np.random.Generator`.

### R7. Конверторы на Windows (без ROCm)
`RawToLfmCube` = дечирп+FFT — это GPU-модули (§6.1). На **Windows/NumpyBackend** нужен numpy-путь:
`heterodyne_base.dechirp_numpy` (есть!) + `np.fft`. → в §4.0 пометить: у каждого `Converter` —
GPU и numpy реализация (Strategy), выбор по платформе.

### R8. Замыкание копий для тестов больше, чем одна строка
Вендоринг `DataValidator` тянет `result.py` (TestResult/ValidationResult), `validators/base.py`,
возможно `reporters.py`. `TestBase`/`SignalTestBase` тянут `gpu_context.py`+`gpu_loader.py`+`configs.py`.
→ в §8 указать **полное замыкание** копируемого (не «один файл»), с атрибуцией на каждый.

## 🟢 Учесть при реализации

### R9. numpy 2.2 (system cp313) vs текущий код
Рантайм — system numpy **2.2.4**. Новый код генераторов пиши сразу под numpy 2.x (без `np.complex_`,
учёт copy-семантики). При миграции всего radar3d (позже) — отдельно прогнать старые модели на 2.2.

### R10. pyyaml на cp313
YAML-конфиг (§9, ответ Q2) вводит зависимость `pyyaml`. Проверить наличие в system cp313; если нет —
это единственная новая зависимость рантайма. `config_source.py` грузит YAML **в существующие `*Spec` VO**
(реюз, не второй конфиг-слой).

### R11. «Всё меняется каждый такт» + «быстрая пересборка» vs Python-first
Q8 (всё динамично) + Q7 (быстрая пересборка, возможно мутация на GPU) плохо стыкуются с примитивом
«перечитывать YAML на такт» (медленно). На фазе P6 — думать про **инкрементальную/GPU-резидентную**
эволюцию сцены, не полную пересборку. Пока — заложить хук `t→spec(t)`, не оптимизировать.

### R12. ФМн: `n_ant` меняется по тактам (Q10) при физически фиксированной решётке 16×16
Вероятно имеется в виду **активная подапертура**. Уточнить на P4/P6 (не блокер).

## 🔴 Пропущено в первом проходе (само-ревью 2026-07-14, сверка с кодом DSP-GPU)

### R13. §6.2 — неверная запись примитивного полинома `0x00400007` 🔴
Спека §6.2 аннотирует `polynomial=0x00400007` как «`x²⁶+x¹+1`, GF(2²⁶)». **Неверно:**
`0x00400007` = биты `{0,1,2,22}` = **`x²²+x²+x¹+1`** (степень **22**, 4 отвода, не трёхчлен).
Сверено с реальным дефолтом DSP-GPU `radar/t_fm_correlator.py:32` (`poly=0x00400007`). GF(2²⁶) дал бы
`0x4000007`. → исправить аннотацию §6.2; на P4 это меняет длину M-последовательности (`2²²−1`, не `2²⁶−1`).

### R14. §8 — `signal_base.py` указан не в том пакете 🟡
Спека §8 копирует `SignalTestBase`/`signal_base.py` «из DSP-GPU `.../common/`». Фактически файл лежит в
**`signal_generators/signal_base.py`** (`class SignalTestBase(TestBase)`, импортит `common.base`/
`common.result`/`common.validators`), в `common/` его **нет**. → поправить путь-источник вендоринга §8.
