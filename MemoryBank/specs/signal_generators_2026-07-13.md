## 0. Что просил Alex (требования, дословно разобрано)

1. Набор **генераторов** сигналов: **АМ**, **ЛЧМ**, **ФМ**.
2. Добавлять к ним **шум**.
3. Полезный сигнал может лежать: **во всём** временном интервале / **в части** / **короткий в любом месте**.
4. Добавлять **помехи как в патенте** + **прочие промышленные** + **радиолюбитель вышел на связь**.
5. Генерация — **двумя способами**: чистый **Python** и **GPU**.
6. На **Windows** — только Python.
7. На **Linux** максимально **переиспользовать** готовое из **DSP-GPU** (реализации + тесты + примеры), использовать (.so) и как примеры реализации.
8. **Не сочинять новые классы** — использовать что есть.

### Расшифровка «ФМ» (согласовано с Alex)
- **ФМн** — фазоманипулированный агильный код 2ⁿ (это **FM-m** зонд патента, гл.6) — наш зондирующий сигнал.
- **ЧМ** — аналоговая частотная модуляция — модель **стороннего источника/помехи** (радиолюбитель/связной).
- → делаем **оба**.

### 🎯 ТЕКУЩИЙ ОБЪЁМ (коррекция Alex 2026-07-13)
- **ТОЛЬКО генерация**: сигнал + шум + помехи на **n×n антенн** → сырое время `[nx, ny, n_samples]`.
- **Куб / дечирп / дальностный FFT (мост время→куб) — НЕ рассматриваем сейчас** (отложено, §6).
- Из DSP-GPU нужны прежде всего **собранные GPU-модели расчёта** (готовые `.so`), а не расчёт куба.

### 🧭 Принцип GPU-first (коррекция Alex 2026-07-13)
> «В конечном итоге всё должно работать на GPU. Python — чисто обвязка + обучение нейронок.»
- Боевой тракт (генерация, токенизатор, инференс) — **GPU** (C++/HIP, собранные модели DSP-GPU).
- **Python = тонкая обвязка** вызовов GPU-моделей **+ обучение сетей** (offline). Не «GPU для тяжёлого» —
  **GPU для всего**; NumPy-ветка — это **эталон + Windows + генерация датасета**, а не основной тракт.

## 1. Ключевое архитектурное решение — расщепление TRAIN / DEPLOY

> Снимает конфликт Python 3.12 ↔ 3.13. Подтверждено проверкой (см. §7).

Конфликт версий был **ложной дилеммой**: он предполагал одну среду для обучения и боя. Разделяем:

```
┌──────────────────────────────┐    веса .npy / .bin    ┌───────────────────────────────┐
│  TRAIN-СРЕДА (cp312)          │ ───── экспорт весов ──► │  RUNTIME (cp313 + C++/HIP)      │
│  • torch-ROCm 2.11 (offline)  │                         │  • dsp_*.so (cp313) — ЕСТЬ ✅   │
│  • учит Gate-MLP + L3-LSTM    │                         │  • генераторы, токенизатор      │
│  • датасет — от НАШИХ ген-ов  │                         │  • инференс: MLP→LDS, LSTM→VRAM │
│  • обучение = НЕ real-time    │                         │  • numpy/scipy — system Debian  │
└──────────────────────────────┘                         └───────────────────────────────┘
```

**Обоснование через патент** (две нейронки — адъюнкты, судья — физика):

| Сеть | Патент | Размер | Память GPU | Роль |
|------|--------|--------|-----------|------|
| **Gate-MLP** `6→16→3` | гл.4.8, п.С.8 | сотни float | **LDS** (on-chip, с токенизатором) | приоритет обработки |
| **L3 LSTM/GRU** | гл.7.3, п.С.7 | единицы КБ | **VRAM** (резидентно) | наводка угол/дальность |

Метку «цель» ставит **физика** (арбитр τ≥0 + код-корреляция FM-m) → сети можно вынести в offline-обучение (cp312), в бою torch не нужен → **runtime свободен для cp313**.

**Контракт передачи весов:** формат `.npy` (портируем 3.12↔3.13) или сырой `float32 .bin` для C++. ONNX избыточен для таких мелких сетей — не берём.

> ⚠️ Это решение **шире задачи генераторов** — здесь зафиксировано как контекст. Детально среду
> обучения проектируем, когда дойдём до сетей (пока — открытый вопрос §11).

## 4. Архитектура слоя генерации (проект)

Единая идея: **сигнал = Waveform (Strategy)** × **Backend (Strategy)** × **Placement (VO Decorator)**;
сборка сцены — существующий **Composite + Factory**. Выход — **абстрактный носитель `SignalField`**
(не голый массив с зашитой семантикой ЛЧМ), эволюция — по **оси тактов**.

### 4.0. Принцип абстрактного представления (правка Alex 2026-07-14)
> «Первоначальный куб (прямоугольник) — **абстрактная вещь**. АМ/ЛЧМ/ФМн/ЧМ представлены **по-разному**;
> в кубе **разные физические данные**. Пост-дечирп куб `(nx,ny,n_real)` — это **только ЛЧМ**.»

- Выход генератора — **НЕ** `[nx,ny,n_samples]` со смыслом «третья ось = бин дальности» (частный ЛЧМ).
- Вводим абстрактный **Value Object `SignalField`**: `data` + **семантика** — тип модуляции,
  **смысл осей**, `fs`, несущая, геометрия `n×n`, индекс такта. Каждая модуляция **наполняет его
  по-своему**; downstream (позже, при кубе/токенах) **диспатчит по типу**.
- «Глубина» куба у разных типов — **разные физ.величины**: ЛЧМ→бин дальности (после дечирпа),
  АМ→позиция скользящего окна, ФМн→задержка корреляции. На этапе **генерации** общий физический
  носитель — сырое время на решётке, но **семантику хранит `SignalField`**, а не соглашение о форме.
- Это прямо ложится на патент: «одно ядро — **сменный интерфейс под тип сигнала**» (гл.0).
- **Конверторы (ответ Alex Q6):** данные формируются **одинаково** (сырое время), а `SignalField`
  несёт семантику. Переход к нужному представлению — отдельные **`Converter`** (Strategy):
  `RawToLfmCube` (дечирп→FFT), `RawToCorrelation` (коррелятор), `RawToAmVolume` (скольз.3D-FFT).
  Конверторы живут **в модуле генератора**; что/куда добавлять — по мере возникновения.
- **R7 (ревью):** у каждого `Converter` — **две реализации** (Strategy): GPU (DSP-GPU `.so`) и numpy
  (Windows/эталон: `dechirp_numpy` + `np.fft`). Выбор — по платформе, как у `Backend`.

### 4.1. Слои

**ТЕКУЩИЙ ОБЪЁМ — только СЛОЙ 1 (генерация на n×n антенн, выход = `SignalField`):**
```
СЛОЙ 1 — ГЕНЕРАЦИЯ  →  SignalField  (НОВЫЙ; носитель абстрактный, payload = сырое время на n×n)
  Waveform (Strategy):  AmWaveform · LfmWaveform · PhaseCodeWaveform(ФМн)   → каждый свой SignalField
  Aperture (реюз):      grid.steering(kx,ky) — вклад по углу на n×n элементов (наш ArrayGrid)
  Помехи-время:         SmspJammer · FmInterferer(ЧМ) · IndustrialRFI · DrfmRepeater · BarrageRF
  Placement (VO):       TimeWindow{ full | partial(t0,t1) | short(t0,dur) }
  Noise:                AdditiveNoise (в getX уже встроен — параметризуем)
  Tacts (заложить):     TactSequence — эволюция сцены от такта к такту (§4.5)
  Backend (Strategy):   HipBackend(GPU, главный) | NumpyBackend(эталон/Windows/датасет)
  Выход:                SignalField{ data, modulation, axes-семантика, fs, несущая, n×n, tact }
```

**ОТЛОЖЕНО (не сейчас, §6):** мост в куб (dechirp/FFT/скольз.окно/коррелятор — по типу сигнала) и
интеграция со старым куб-слоем. Сейчас — только наполнение `SignalField` (payload = сырое время).

### 4.2. Новые абстракции (минимум сущностей)
| Абстракция | Паттерн | Ответственность (SRP) |
|-----------|---------|----------------------|
| **`SignalField`** (VO) | Value Object | **абстрактный носитель**: `data` + семантика (тип модуляции, смысл осей, `fs`, несущая, `n×n`, такт). Каждый тип наполняет по-своему |
| `Waveform(ABC/Protocol)` | Strategy | `.render(backend, spec) → SignalField` (свой для каждой модуляции) |
| `GenBackend(Protocol)` | Strategy | примитивы `phase/exp/window/noise` на NumPy **или** HIP |
| `TimeWindow` (VO) | Value Object + Decorator | маска размещения (full/partial/short), оборачивает Waveform |
| `WaveformSpec` (VO) | Value Object | чистые данные (fs, f0, fdev, amp, phase, tau, окно, **`snr_db`** — R5) — как `LfmParams` |
| `TactSequence` | Iterator/Builder | эволюция сцены по тактам (движение цели, помехи во времени) — §4.5 |
| ~~`TimeToCubeBridge`~~ | Facade | ⏸️ отложено — мост в куб вне текущего объёма (§6) |

> **Реюз паттернов:** `Waveform` зеркалит существующий `ISignalStrategy` (DSP-GPU) и `SignalSource`
> (наш). Фабрику берём **существующую** `EmitterFactory` (расширяем реестр новыми spec), Composite —
> существующий `Scene`. Новых фабрик/композитов НЕ плодим.

### 4.3. Два бэкенда за одним интерфейсом (SOLID: DIP + LSP)
```
GenBackend(Protocol):
    exp_phase(phase) · make_window(mask) · add_noise(x, power, seed)
HipBackend   — ГЛАВНЫЙ (боевой): собранные GPU-модели DSP-GPU (dsp_signal_generators, cp313)
NumpyBackend — эталон + Windows + генерация датасета (portable; работает в cp312 и cp313)
```
- **GPU-first**: боевой тракт — `HipBackend` (Python лишь зовёт GPU-модель). `NumpyBackend` —
  эталон/Windows/датасет, **не** основной тракт.
- **Windows** → всегда `NumpyBackend` (выбор по платформе в Composition Root).
- **Ядро Waveform version-agnostic** (чистый numpy для эталона) → импортируется и в cp312, и в cp313.
- torch для генерации **НЕ используется** (elementwise; torch — только обучение сетей).

### 4.4. Размещение полезного сигнала (требование §0.3)
`TimeWindow` (VO) — маска по временной оси, как `in_window` в `getX_numpy`:
- `full` — весь интервал `[0, T)`;
- `partial(t0, t1)` — кусок;
- `short(t0, dur)` — короткий импульс в любом месте.
Применяется **декоратором** поверх любого `Waveform` (не дублируем в каждом классе).

**R5 (ревью) — калибровка по SNR:** источник задаётся через **`snr_db`** (датасет весь в дБ,
industrial §8). Амплитуда сигнала считается от мощности шума: `A=√(σ²·10^(SNR/10))`; для сверки —
реюз `core/snr`. **R6 (ревью) — детерминизм:** вся случайность (шум, код, α-stable помехи, tacts) —
через **переданный `np.random.Generator`** (seed в конфиге), чтобы датасет воспроизводился.

### 4.5. Ось тактов — движение и время (заложить, правка Alex 2026-07-14)
> «Заложить и рассмотреть **передвигающуюся модель от такта к такту** + **изменение помех по времени**.»
- Сцена — не один снимок, а **последовательность тактов** (кадров): между тактами **цель движется**
  (дальность/угол/скорость), **помехи меняются** (агильный код, включение/выключение, дрейф).
- `TactSequence` выдаёт `SignalField` на такт `t`; параметры источников — функция `t` (эволюция спеки).
- Нужно для будущего: трекинг «летит» (§Доплер), обучение LSTM на потоке токенов вдоль тактов.
- **Реализация**: один такт — первым; движок эволюции (`t→spec(t)`) — сразу закладываем в API,
  включаем следующим шагом. Так `SignalField` уже несёт `tact`-индекс (§4.2).
- **Ответ Alex Q8:** от такта к такту меняется **ВСЁ** — система постоянно динамична (сигнал, код,
  помехи, положение). Отсюда жёсткое требование — **модель должна быстро пересобираться** (Q7).
- **Ответ Alex Q7 (движение — на потом):** примитивный вариант — новая «прорисовка» через YAML на такт;
  для АМ — стирать/перезаписывать размытый прямоугольник. ⏳ **Кодо: изучить, как моделируют движение
  другие (range-walk, доплер, marching), предложить варианты**; условие — быстрая пересборка,
  возможна мутация прямо на GPU. Числа/механизм — обсудим, когда дойдём.

### 4.6. Разделение ответственности (GRASP, ответ Alex Q4)

> «GPU-DSP генератор генерит **в заданной антенне заданный сигнал+помехи**. Расположение помех и
> объектов в пространстве создаёт **другая программа** моделирования. Чистый GRASP.»

| Ответственность | Кто | Знает |
|-----------------|-----|-------|
| **Пространственная сцена** (где цель/помехи, углы, дальности, движение) | **`SceneModeler`** (новый) | геометрию, стиринг, эволюцию по тактам |

SceneModeler **задаёт** параметры (угол/задержка/амплитуда на элемент) → Generator **синтезирует**.

| Сигнал | Источник формулы | Статус |
|--------|-----------------|--------|
| **CW** (опора) | `cw_numpy` (DSP-GPU) | вендорим |
| **ЛЧМ** | `lfm_numpy` / `getX_numpy` (DSP-GPU) | вендорим |

## 6. Готовые GPU-интерфейсы DSP-GPU по типу сигнала (реюз)

> Каждый тип сигнала — свой готовый GPU-модуль DSP-GPU. Не строим свои: подаём наш
> сигнал, результат пишем в **нашу матрицу** (`SignalField`). Патент: «сменный интерфейс».

### 6.1. ЛЧМ — дечирп (готов)

> Для **ЛЧМ** дечирп (гетеродин) в DSP-GPU **уже реализован**.

**Готовый модуль:** `dsp_heterodyne.HeterodyneROCm` (cp313, импорт подтверждён).
Пример/эталон: `DSP/Python/heterodyne/` (`heterodyne_base.py`, `t_heterodyne_rocm.py`).

**API (сверено):**
```
het = HeterodyneROCm(ctx)
dc  = het.dechirp(rx, ref)        # rx[n_ant,n_samples], ref[n_samples] → dc[n_ant,n_samples]
# numpy-эталон:  dc = rx * conj(ref)      beat_freq = chirp_rate · задержка
```

**Range-FFT (готов):** `dsp_spectrum.FFTProcessorROCm.process_complex(data, sample_rate=fs)` —
**zero-pad до 2^n встроен** (`nFFT = nextPow2`), батч по лучам `(beam_count, nFFT)`.
Пример: `DSP/Python/integration/t_signal_to_spectrum.py`.

**Цепочка ЛЧМ (весь тракт — реюз готовых GPU-модулей):**
```
наш генератор → SignalField(ЛЧМ, сырое время [nx,ny,n_samples])
      │  reshape → rx [n_ant=nx·ny, n_samples]
      ▼
HeterodyneROCm.dechirp(rx, ref)            ← ГОТОВО (дечирп)
      │  дечирпованный [n_ant, n_samples]
      ▼
FFTProcessorROCm.process_complex(dc, fs)   ← ГОТОВО (range-FFT + zero-pad→2^n)
      │  [n_ant, nFFT]  →  reshape
      ▼
куб (nx, ny, nFFT)  →  пишем в нашу матрицу (SignalField, kind=cube)
```

- Весь ЛЧМ-тракт генератор→дечирп→range-FFT→куб собирается из **готовых** DSP-GPU-модулей.
- Интеграция куба со старыми моделями (3FFT/nuller/CFAR) — по-прежнему отдельный шаг (не сейчас).
- ⚠️ Нужен **опорный `ref`** (эталонный ЛЧМ такта) — наш же генератор его и даёт (§11 п.9).

### 6.2. ФМн — коррелятор M-последовательности (готов, правка Alex 2026-07-14)

> Коррелятор в DSP-GPU **готов**. **Генерацию M-последовательности переносим в НАШИ генераторы**
> (`PhaseCodeWaveform`), а код подаём в готовый коррелятор.

**Готовый модуль:** `dsp_radar.FMCorrelatorROCm` (cp313, импорт подтверждён).
Пример/эталон: `DSP/Python/radar/t_fm_correlator_rocm.py`.

**API (сверено):**
```
corr = FMCorrelatorROCm(ctx)
corr.set_params(fft_size, num_shifts=K, num_signals=S, num_output_points=n_kg, polynomial, seed)
corr.prepare_reference_from_data(ref)    # ← НАШ код (m-последовательность)
peaks = corr.process(input)              # [S, K, n_kg] — пики корреляции (ось задержки)
# внутренний LFSR: polynomial=0x00400007 (x^22+x^2+x^1+1, GF(2^22), степень 22, длина 2^22−1) — для нашего кода задаём свой
#   ⚠️ сверено с DSP-GPU radar/t_fm_correlator.py:32 (биты {0,1,2,22}); НЕ x^26/GF(2^26)
```

**Цепочка ФМн:**
```
наш генератор m-послед. (LFSR) → код 2ⁿ  →  prepare_reference_from_data(ref)
наш ФМн-сигнал (SignalField) ────────────►  process(input) → [S,K,n_kg]
                                                   │  пики корреляции = задержка/дальность
                                                   ▼
                                          пишем в нашу матрицу (SignalField, kind=correlation)
```

- Генерация кода — **наша** (переносим из коррелятора в `PhaseCodeWaveform`, §5), коррелятор — **реюз**.
- Это и есть «ФМн→задержка корреляции» из §4.0 — представление **иное**, чем куб ЛЧМ.
- **Ответ Alex Q10:** полином — **примитивный по умолчанию**; от такта к такту меняем **всё** —
  длину кода, число корреляторов (`K`), глубину по дальности (`n_kg`), число антенн.
