# Словарь дальностных величин: развязка «метров на бин» (затык из чата 3D-FFT body motion)

**Дата:** 2026-07-15 · **Причина:** в чате «3D-FFT body motion» смешались **три разные величины**,
все размерности «метры», отсюда путаница (12.5 м/бин vs 25 м/бин vs «2 бина = 50 м = Δr»).
Проверено численно (numpy, симуляция дечирп→FFT, R_true=6000 м) — все формулы ниже сошлись.

---

## 1. Три величины — НЕ путать

| # | Величина | Формула | Что это | От чего зависит |
|---|----------|---------|---------|-----------------|
| **V1** | Шкала (метров на бин FFT) | `c·f_s / (2·μ·N_fft)` | пересчёт «номер бина → дальность»: `R = k·V1` | от **N_fft** (с zero-pad уменьшается!) |
| **V2** | Разрешение Δr (ширина пика) | `c / (2·μ·T_overlap) = c/(2·ΔF_eff)` | физическая ширина всплеска в метрах | от **захваченной девиации**, НЕ от N_fft |
| **V3** | Строб сырого отсчёта | `c / (2·f_s)` | метры на отсчёт **быстрого времени ДО дечирпа** (интерпретация «время пролёта») | после дечирпа **не применима вообще** |

**Связка:** `V1 = V2` только при `N_fft = N` **и** полном перекрытии эха с опорой.
`V1 = V3` только при `ΔF = f_s` (у нас не так). Это три разных объекта.

## 2. Численная проверка (fs=12 МГц, ΔF=6 МГц, N=1024, R=6000 м)

```
без пада (Nf=1024):  пик бин 784,  R=6000 м, шкала 25.00 м/бин, ширина −3дБ ≈ 25 м*
пад ×4  (Nf=4096):   пик бин 3136, R=6000 м, шкала 6.25 м/бин,  ширина −3дБ ≈ 44 м
Δr номинал  c/(2·ΔF)          = 25.0 м
Δr эффект.  c/(2·μ·T_overlap) = 47.1 м   (эхо перекрывает опору на 53%)
строб       c/(2·fs)          = 12.5 м   ← это НЕ шкала после дечирпа
```
\* без пада ширина «25 м» — иллюзия грубой сетки (1 бин); zero-pad лишь **прорисовал**
реальную ширину 44–47 м, он её не создал.

## 3. Разбор нестыковок соседнего чата

- **«dr = c/(2·fs) = 12.5 м/бин»** (начало чата) — ошибка категории: V3 применили как шкалу
  после дечирпа. Правильная шкала — V1.
- **«длина 2 бина = 50 м = Δr»** — арифметика не с номиналом: Δr_номинал = 25 м. Но физика
  была права случайно: 50 м ≈ Δr_эфф = 47 м, потому что эхо перекрывало опору лишь ~наполовину.
- **«ширина ~44 м, чуть шире Δr=25 м, т.к. эхо перекрывает ~половину записи»** — вывод верный,
  теперь формализован: `Δr_eff = c/(2·μ·T_overlap)`; при 53% перекрытия = 47.1 м. Совпало.
- **Отрицательная f_b** — не баг: при `dechirp = rx·conj(ref)` тон = `exp(−j2πμτt)` → `f_b = −μτ`.
  Конвенция: либо `ref·conj(rx)` (плюс), либо маппить по `|f|`/fftshift — но зафиксировать ОДНУ.

## 4. Канонический рецепт для P5 (LfmToCube)

```
1. дечирп:      s = rx · conj(ref)                (знак f_b = −μτ, учесть при маппинге)
2. zero-pad:    N → N_fft = next_pow2(N)·(1|2|4)   (прорисовка пика; окно по дальности НЕ ставить
                                                    до измерения ширины — Хэмминг уширит ×~1.5)
3. FFT, шкала:  R(k) = k_signed · c·f_s/(2·μ·N_fft)   ← ЕДИНСТВЕННАЯ формула бин→метры
4. ширина цели: ожидание = c/(2·μ·T_overlap) / V1 бинов; критерий «точечная» сравнивать
                с Δr_eff, а НЕ с номиналом c/(2·ΔF)
```

**Тест-критерии P5 (уточнение приёмки):**
- `|R_est − R_true| ≤ V1/2` (полбина шкалы);
- ширина −3дБ = `c/(2·μ·T_overlap)` ± 20% (rect, без окна);
- при полном перекрытии (τ≪T): ширина → `c/(2·ΔF)` = 25 м;
- инвариант: zero-pad ×4 меняет V1 в 4 раза, **ширину в метрах не меняет**.

## 5. Связь со спекой заявки

Дополняет `LFM_dechirp_range_peak_2026-07-15.md` (§4.2 padding): туда же добавлен фактор

## 6. Уточнение §4 (численная проверка 2026-07-15, Кодо)

Инвариант «zero-pad ширину в метрах не меняет» — **не буквально**: измеренная −3дБ ширина
**сходится снизу** к `Δr_eff` по мере роста N_fft (грубая сетка недо-разрешает лепесток):
```
N_fft=1024 -> 25.0 м (1 бин, иллюзия) ; 4096 -> 37.5 м ; 8192 -> 40.6 м  → Δr_eff=47.1 м
R_est: 5995.8 / 6002.1 / 5999.0 м (ошибка < V1/2 всегда) ✓
```
Физическая ширина фиксирована (47 м), N_fft только «прорисовывает». Критерий ширины в P5
мерить при **N_fft ≥ 4·N** (сетка достаточно тонкая), сравнивать с `Δr_eff`, не с сырым 1-бином.
