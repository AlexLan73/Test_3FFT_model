## 1. Что меняем (суть)

Сейчас источник (`run_history`) **вшит** в сервер. Разделяем на два слоя:

```
[ИСТОЧНИК реалтайм]  →  PanelPublisher.push_tick(payload)  →  WebSocketTransport  →  браузер
  (твои данные)          (тонкая обёртка, НОВОЕ)              (готово)              (готово)
```

- **`PanelPublisher`** (НОВЫЙ, тонкий): `start(port)`, `push_meta(...)`, `push_tick(index, payload)`,
  `close()`. Внутри — `WebSocketTransport` + буфер `meta`/последних тактов для поздних клиентов.
- **Мост «твои данные → tick»**: функция, превращающая твой формат в контракт §2.
- Источник вызывает `push_tick(...)` по мере поступления данных (реалтайм-цикл).

## 2. Контракт данных (то, что рисует панель — НЕИЗМЕННО)
### 2.1. Этот вариант для отладки подойдет
Панель понимает ровно это (мировые координаты kx,ky,r — проекция на клиенте единой камерой):

### `meta` (один раз в начале сессии + при смене апертуры)
```jsonc
{
  "nx": 64, "ny": 64, "nAxis": 4096, "kTrail": 8, "nTicks": <всего|null для бесконечного>,
  "sig": "lfm",                   // тип сигнала по умолч. → фронтенд обработки (§3 Этап B, патент гл.2-3/4-бис)
  "stats": {...},                 // строки статуса (опц.)
  "finalFeats": {"<id>": {"label":..., "f": {6 признаков §4.11}}},   // для таблицы
  "stations": [{"type":"radar|ham", "label":..., "kx":..., "ky":...}],
  "cam": <Projection.as_js()>     // единая камера core (az0/el0/fit, field.az=π)
}
```

### `tick` (на каждый такт)
```jsonc
{
  "truth": {"t":[kx,ky,r], "c":[kx,ky,r], "b":[kx,ky]},   // истина цель/носитель/barrage (опц.)
  "band":  [kx,ky] | null,                                // полоса barrage → null
  "pts":   [[kx,ky,pos,db], ...],                         // точки-детекции (turbo по дБ)
  "trk":   [{"id","kx","ky","mv","h":[[kx,ky],...]}, ...],// треки + след (h ≤ kTrail)
  "sl":    [{"id","kx","ky","pos","mv","x0","y0","m":[[дБ]...]}, ...],  // срезы (кроп ±8)
  "feats": {"<id>": {6 признаков §4.11}} | null  // ОПЦ.(отладка): по-тактные признаки → таблица без пересчёта (§4.1)
}
```

> Любой источник, формирующий этот `tick`, работает с панелью без её изменений.

### 2.2. Рабочий (боевой) вариант — сырьё → обработка → `tick`

Панель по-прежнему получает **только** `tick` §2.1. Меняется то, что стоит **до** неё:
источник отдаёт не готовые примитивы, а **сырой потактный объём по всем антеннам**
(сигнал + шум + помехи), а `tick` формируется обработкой на приёмной стороне.

**Поток данных:**
```
[ГЕНЕРАТОР сырья на GPU]                 [ПРИЁМ + ОБРАБОТКА]              [ПАНЕЛЬ]
 сырой куб nx×ny×nAxis complex   ──►   6-порог. обработка (GPU/CPU)  ──►  tick §2.1  ──► рисует
 (по каждой антенне: сигнал+шум+помехи)  → разрежённая матрица (без шума)
                                          → лёгкий tick (~6 КБ), лог хранит все такты сессии
```

**Источник сырья (RawCubeSource, Strategy — реализации равнозначны):**
- **`SocketSource`** — генератор сырья на **отдельном GPU-компьютере**, кадры идут по сети
  (ZMQ/1 Гбит) → `ZmqTransport`. Боевой реалтайм-режим больших тестов.
- **`FileSource`** — большой записанный набор потактно на диске (сырьё с полезным сигналом);
  читается через **Observer** с нужным интервалом. Оффлайн-реплей.
- **Отладка:** генератор сырья — на **том же** GPU (или во временной Python-заглушке),
  малые размеры, без сети.

**Размер кадра:** тест `64×64×4096` complex64 = **128 МБ/такт**; худший `512×256×10000` ≈ **10 ГБ/такт**
(§4.5). По сети сырьё реалтаймит только на малых размерах — иначе обработка на GPU-источнике, по сети
лёгкий `tick`. В браузер сырьё **не шлём** (обработка на приёмной стороне; в браузер идёт лёгкий `tick`).

**Обработка (реюз тракта `core` ex4):** сырой куб → **фронтенд по типу сигнала `sig`** (для ЛЧМ —
дечирп + дальностный/угловой FFT; см. §3 Этап B, патент гл.2–3) → детекция (6-порог) → белый шум
отбрасывается, остаётся **разрежённая матрица только сигналов** → `pts` + `trk`/`sl` §2.1.
Такт лёгкий (~6 КБ); **весь лог сессии храним** (1000 тактов ≈ 6 МБ), `kTrail=8` — лишь дефолт
длины следа на отрисовке (панель может показать весь путь). Детали — §4.1.

**Итог:** после обработки сырья получаем ровно контракт §2.1 → панель рисует без изменений.
Соответствие этапам: `SocketSource` → **Этап C**, `FileSource` → **Этап D**, обработка сырья на
сервере → **Этап B**.

### 🔁 Общий тракт обработки сырья — Этапы B · C · D (вариант 1)

> Решение Alex (2026-07-20): **B, C, D различаются ТОЛЬКО источником/транспортом сырья.**
> Обработка — **одна**, в `core`, не дублируется. Меняем источник — тракт не трогаем (OCP).

```
сырой куб nx×ny×N complex + sig
   → SignalFrontend[sig]      (Strategy: сменный фронтенд по типу сигнала)
   → общее ядро               (детекция 6-порог → band/null → трекер → срезы)
   → tick §2.1                → publish → панель рисует
```

**Тип сигнала — сменный фронтенд (Strategy).** Патент: одно ядро, но **разные интерфейсы под тип
сигнала** — сырой куб нельзя обработать одинаково. Кадр несёт `sig`; сервер выбирает `SignalFrontend`:

| `sig` | Фронтенд (доп. обработка ДО детекции) | Патент | Результат |
|-------|--------------------------------------|--------|-----------|
| **`lfm`** (ЛЧМ) | **дечирп** (stretch, если не на ПЛИС) → **дальностный FFT** (частота→дальность) → **угловой FFT** → куб угол×угол×дальность | гл.2 (тракт), гл.3 (два FFT) | точность по дальности |
| **`am`** | AM-фронтенд (грубое присутствие) | гл.4-бис | быстрое присутствие |
| прочие | добавляются подклассом `SignalFrontend` (OCP, тракт не трогаем) | — | — |

- **ЛЧМ:** «добавить доп. обработку и обновить данные» = дечирп → дальностный FFT `f_b=μ·2R/c` →
  угловой FFT (данные держим **комплексными** до углового FFT — нужна фаза). Только потом
  6-порог/детекция → `tick`. Формулы — [`specs/LFM_dechirp_range_peak_2026-07-15.md`], патент гл.2–3.
- **Общее ядро** (детекция → band/null → трекер → срезы → `tick`) — **после** фронтенда, одно на все `sig`.
- Фронтенды — в `core/` (математика, правило 07), выбор по `sig` — **Abstract Factory/Registry**
  (как `EmitterFactory`); панель о типе сигнала не знает.
- ⚠️ Размер куба: тест `64×64×4096` = **128 МБ/кадр**, худший `512×256×10000` ≈ **10 ГБ/кадр** (§4.5) —
  в браузер сырьё не шлём: обрабатывается на СЕРВЕРЕ/GPU-источнике, в браузер лёгкий `tick`+срезы.

> Ниже у B/C/D описан **только источник/транспорт** — обработка везде эта.

### Этап B — источник Python (`push_cube`)
- `push_cube(index, cube, sig)` → общий тракт ↑. Сырьё генерит Python/тот же GPU, малые размеры, без сети.

### Этап C — источник C++/GPU (ZMQ)
- Внешний процесс C++/HIP генерит сырьё на **отдельном GPU** → `ZmqTransport` (приём) →

### Этап D — источник файлы/реплей (Observer)

## 4. Открытые вопросы (ОБСУДИТЬ ПО ШАГАМ, до кода)

1. **Живой край vs история — ✅ РЕШЕНО (Alex, 2026-07-20): храним ВСЮ историю сессии; границы = старт/стоп потока.**

   Уточнение Alex: после обработки на сервере у нас **не куб, а лёгкий разрежённый список координат**
   объектов (детекции/треки/срезы §2.1). Он крошечный:
   - оценка Alex: теплокарта токенов 1 объекта ≈ 300 Б; 20 объектов ≈ **6 КБ/такт**;
     **1000 тактов ≈ 6 МБ** — ничтожно. ✅ Арифметика верна (300·20·1000 = 6·10⁶ Б).
   - ⇒ **кольцевой буфер НЕ нужен** — держим весь лог сессии, память не проблема.

   **Границы истории — старт/стоп потока данных** (а не окно `N`): «старт» открывает сессию
   (новый лог + `meta`), «стоп» замораживает. Слайдер перематывает **весь** лог сессии.

   **Хвост трека `kTrail=8` — это только ДЕФОЛТ отрисовки, не предел хранения:**
   - по умолчанию рисуем последние 8 тактов движения;
   - тумблер «**весь путь**» — панель рисует всю траекторию объекта (реконструирует из своего буфера
     тактов, который и так полный; сервер тоже держит полный лог). **Чисто клиентский тумблер.**

   **Проектирование (SOLID / GRASP / GoF):**
   - `Tick` — **Value Object** (иммутабельный лёгкий payload §2.1, `frozen`).
   - `TickLog` — **Pure Fabrication** + Информационный эксперт: **append-only лог ОБРАБОТАННЫХ
     тактов** (лёгких `tick`, храним ВСЁ), отдаёт снапшот/срез. Открытие/закрытие = старт/стоп.
     (Опц. предохранитель — верхний потолок тактов в конфиге; по умолчанию не режем.)
   - `PanelPublisher` — зависит от `TickLog` через **DI**; клиенты — подписчики (**Observer**).
     Поздний клиент (§4.4): на `on_connect` → реплей `meta` + весь лог сессии (или хвост — конфиг).

   **⚠️ Два разных ПЛАНА данных (уточнение Alex, 2026-07-20) — не путать:**

   | План | Что | Размер | Хранение |
   |------|-----|--------|----------|
   | **Сырьё** — `RawQueue` (Observer) | сырой куб с GPU/SSD | большой (~64 МБ/такт) | **транзит: consume-and-drop** — забрали на обработку → удалили; опц. архив блоками на диск для повторного прогона |
   | **Обработанное** — `TickLog` | лёгкий `tick` §2.1 | ~6 КБ/такт | **храним ВСЁ** (1000 ≈ 6 МБ) |

   «Храним всё» — это про **лёгкий `TickLog`**, а НЕ про сырьё. Сырьё в Observer не копим (велико):
   удаляем по мере забора (опц. — блоками на диск для повторного прогона). ✅ Как и предложил Alex.

   **Развязка producer↔consumer (проблема «рендер медленнее потока»).** Обработка пишет `tick`
   в `TickLog` **по мере расчёта** (producer); панель читает из `TickLog` **в своём темпе** (consumer).
   Поток/обработка и визуализация развязаны: панель может отстать, встать на паузу для анализа таблиц —
   данные не теряются (всё в логе). Это ровно «писать в отдельный список» из реплики Alex.

   **Три состояния потока — `start` / `stop` / `pause`:**
   - **`start`** — открыть сессию: приём сырья + новый `TickLog` + `meta`. Инициатор — ЛИБО источник
     (удалённый GPU начал слать → мы приняли), ЛИБО панель (настроили директорию → «старт»).
     Закладываем **оба**; контракт — два параметра `start`/`stop` (§4.2).
   - **`stop`** — закрыть сессию: остановить приём, заморозить лог.
   - **`pause`** — **клиентская**: панель замирает на такте (читать таблицы/анализ), источник и
     обработка продолжают наполнять `TickLog`. Возобновление — доиграть с места или прыгнуть на край.
   - **Что показывать (живой край / пауза / весь путь) — состояние КЛИЕНТА** (тонкий слой показа,
     правило 07). Сервер публикует край; `pause`/«весь путь» локальны панели, спец-протокола не требуют.

   **Таблицы при перемотке/паузе (📌 пометка на отладку).** Чтобы таблицы строились для ЛЮБОГО такта
   без пересчёта — заложить в `tick` **по-тактные значения признаков** (опц. поле `feats`), из которых
   строится таблица. Тогда пауза на любом такте показывает корректную таблицу прямо из лога.

2. **Обратный канал браузер→сервер — ✅ решено (§4.1): закладываем ОБА инициатора.** Контракт потока —
   **два параметра `start`/`stop`** (+ клиентская `pause`). Инициатор `start`/`stop`: ЛИБО источник
   (удалённый GPU начал слать), ЛИБО панель (кнопка). Значит канал браузер→сервер **нужен** (минимум
   под `start`/`stop`; `WebSocket` двусторонний — расширяем на команды). `pause`/«весь путь»/выбор
   трека — клиентские, канал не нужен.
3. **Backpressure — ✅ уточнено §4.1 (два плана).** **Сырьё** (`RawQueue`): источник быстрее обработки →
   дроп/архив блоками (память не копим). **`TickLog`** (обработанное) — храним всё; рендер медленнее
   потока не теряет данных (панель читает в своём темпе, перемотка полная). Очередь под tick не нужна.
4. **Поздний клиент — ✅ закрыто §4.1.** `on_connect` в `WebSocketTransport` → реплей `meta` + лог

5. **Частота/размер — ✅ ориентиры (Alex, 2026-07-20).**
   - **Отладка/тест:** куб `64×64×4096` complex64 = **128 МБ/такт** (16.8 млн компл. × 8 Б).
     (⚠️ ранее в спеке стояло «64 МБ» — арифметическая ошибка ×2, исправлено.)
   - **Худший случай:** `512×256×10000` complex64 ≈ **10 ГБ/такт** (1.31 млрд × 8 Б). Остальные меньше.

   ⚠️ **Следствие для сети (важно — определяет вариант передачи):** сырьё худшего случая по 1 Гбит
   (~125 МБ/с) = **~84 с/такт** — НЕ реалтайм; даже отладочные 128 МБ ≈ 1 с/такт. Вывод:
   - **сырой куб по сети (вариант 1, Этап C-raw)** — только для **малых/отладочных** размеров или
     того же компьютера (без сети);
   - для **больших тестов** сырьё **обрабатывается на GPU-источнике**, по сети идёт лёгкий `tick`
     (**вариант 2, Этап C-tick**). В браузер `tick` лёгкий при ЛЮБОМ размере куба.

   **FPS:** целевой пока не задан → параметр (дефолт напр. 10–30 к/с на отрисовке); на живом крае
   при отставании — дроп кадров показа (§4.3), лог `TickLog` не теряет.
6. **«Истина» (`truth`) — ✅ поясняю и решаю.** `truth` в §2.1 = **эталонные истинные позиции**
   цели/носителя/barrage, которые знает **СИМУЛЯТОР** (мы сами сгенерили сцену). В demo ex4 их рисуют
   поверх детекций — сверить, куда попал детектор. Это НЕ «правда мира», а «что знает генератор сцены».
