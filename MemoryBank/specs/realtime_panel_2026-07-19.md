## 1. Что меняем (суть)

Сейчас источник (`run_history`) **вшит** в сервер. Разделяем на два слоя:

```
[ИСТОЧНИК реалтайм]  →  PanelPublisher.push_tick(payload)  →  WebSocketTransport  →  браузер
  (твои данные)          (тонкая обёртка, НОВОЕ)              (готово)              (готово)
```

- **`PanelPublisher`** (НОВЫЙ, тонкий): `start(port)`, `push_meta(...)`, `push_tick(index, payload)`,
  `close()`. Внутри — `WebSocketTransport` + буфер `meta`/последних тактов для поздних клиентов.
- **Мост «твои данные → tick»**: функция, превращающая твой формат в контракт §2.
- Источник вызывает `push_tick(...)` по мере поступления данных (реалтайм-цикл).

## 2. Контракт данных (то, что рисует панель — НЕИЗМЕННО)

Панель понимает ровно это (мировые координаты kx,ky,r — проекция на клиенте единой камерой):

### `meta` (один раз в начале сессии + при смене апертуры)
```jsonc
{
  "nx": 64, "ny": 64, "nAxis": 4096, "kTrail": 8, "nTicks": <всего|null для бесконечного>,
  "stats": {...},                 // строки статуса (опц.)
  "finalFeats": {"<id>": {"label":..., "f": {6 признаков §4.11}}},   // для таблицы
  "stations": [{"type":"radar|ham", "label":..., "kx":..., "ky":...}],
  "cam": <Projection.as_js()>     // единая камера core (az0/el0/fit, field.az=π)
}
```

### `tick` (на каждый такт)
```jsonc
{
  "truth": {"t":[kx,ky,r], "c":[kx,ky,r], "b":[kx,ky]},   // истина цель/носитель/barrage (опц.)
  "band":  [kx,ky] | null,                                // полоса barrage → null
  "pts":   [[kx,ky,pos,db], ...],                         // точки-детекции (turbo по дБ)
  "trk":   [{"id","kx","ky","mv","h":[[kx,ky],...]}, ...],// треки + след (h ≤ kTrail)
  "sl":    [{"id","kx","ky","pos","mv","x0","y0","m":[[дБ]...]}, ...]  // срезы (кроп ±8)
}
```

> Любой источник, формирующий этот `tick`, работает с панелью без её изменений.

## 3. Этапы (от простого к сложному) — ОБСУДИМ КАЖДЫЙ

### Этап A — вариант 2 + Python (ПРОСТЕЙШИЙ, первый) 🎯
**Ты в Python формируешь готовые примитивы, панель рисует.**
- `PanelPublisher` (новый класс в `core/runtime/` или `demo/`).
- `demo/ex4_flight/live_demo.py` — реалтайм-цикл: заглушка-источник (например, летящая точка
  по простой траектории) → `push_tick(готовый payload)` с реальной паузой (`time.sleep`).
- Панель показывает **живой поток**, а не предрасчёт. Доказывает реалтайм end-to-end.
- **Объём:** маленький (обёртка + пример). **Риск:** нет.

### Этап B — вариант 1 + Python (сырой куб → обработка на сервере)
**Ты шлёшь СЫРОЙ объём (nx×ny×N complex), сервер считает детекцию/трекинг/срезы.**
- `push_cube(index, cube)` → сервер прогоняет лёгкий конвейер (реюз ex4: `coarse_burst_points`
  → band/null → трекер → срезы) → формирует `tick` → publish.
- Реюз готового тракта `core` (токенизатор/трекер) — математика в core, не дублируем.
- ⚠️ **ОБСУДИТЬ:** размер куба 64×64×4096 complex64 = **64 МБ/кадр** — по WS в браузер тяжело.
  Решения: (а) куб обрабатывается на СЕРВЕРЕ, в браузер идёт только лёгкий `tick`+срезы (реко);
  (б) урезать куб; (в) слать куб только по запросу. Скорость обработки — на CPU (GPU позже).

### Этап C — вариант 1/2 + C++/GPU (другой процесс) 
**Источник — внешний процесс (C++/HIP), шлёт кадры через ZMQ (`codec.py` язык-нейтрален).**
- Сервер-**мост**: `ZmqTransport` (приём от C++) → `FanOutTransport` → `WebSocketTransport` (браузер).
- C++ формирует либо готовый `tick` (вариант 2), либо сырой куб (вариант 1 → сервер обрабатывает).
- Реюз `FanOutTransport` (уже есть). ⚠️ **ОБСУДИТЬ:** формат от C++ — msgpack по схеме codec.

### Этап D — из файлов/записей
**Источник — файлы приёмника/датасет, читаются по мере появления.**
- `FileSource` (watch-папку/поток записей) → `push_tick`. Для оффлайн-воспроизведения записей.

## 4. Открытые вопросы (ОБСУДИТЬ ПО ШАГАМ, до кода)

1. **Живой край vs история.** Реалтайм-поток бесконечный. Панель сейчас буферизует всё (перемотка
   слайдером). Для реалтайма: держать всю историю (память растёт) или окно последних N тактов?
   Гибрид: живой режим (следим за краем) + кнопка «пауза» фиксирует буфер?
2. **Обратный канал браузер→сервер.** `WebSocketTransport` сейчас **publish-only** (нет команд от
   браузера). Нужны ли команды из панели (пауза/выбор трека/добавить цель)? Если да — расширять.
3. **Backpressure.** Если источник быстрее рендера/сети — дропать старые кадры (как PUB slow-joiner)
   или очередь? Для живого превью дроп норм.
4. **Поздний клиент.** Подключился в середине — сервер шлёт `meta` + текущий снапшот сразу
   (нужен хук on_connect в `WebSocketTransport` — сейчас `meta` идёт в начале цикла).
5. **Частота/размер.** Целевой FPS? Размер куба (для варианта 1)? От этого — трафик и дроп-политика.
6. **Кто владелец «истины».** В реалтайме от приёмника истины (truth) нет — только детекции.
   `truth`/`band` в `tick` делаем опциональными (панель уже терпит null).
7. **meta динамическая.** Апертура/станции меняются в сессии? Если да — `push_meta` повторно.
