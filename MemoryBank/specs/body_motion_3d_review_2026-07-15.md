## 🔴 Критические (расхождение спеки с реальным кодом)

### F1 — `DataContext` в коде ≠ «базовый класс всего обмена» из спеки
`core/data_context/data_context.py` — **тонкий Facade** над `CubeRepository`:
только `save_cube(name, cube)` / `load_cube(name)`. Ни `publish`/`subscribe`, ни
Observer/Subject, ни Blackboard, ни состояния сцены — **ничего** из §4 спеки.

Спека молча предполагает **полную переделку** существующего класса — это «менять
архитектуру без явного согласования» + смешение персистентности (Repository) и
рантайм-шины (pub/sub) в одном классе **ломает SRP**, который сама же спека хвалит.

**Фикс (внесён):** `DataContext` **расширяем, не переписываем** — сохраняем
`save_cube`/`load_cube` (обратная совместимость). Шину (Subject: `publish`/`subscribe`/
`notify`) выносим в отдельный `Blackboard`/`MessageBus`, а `DataContext` его
**композирует** (persistence и pub/sub — разные ответственности).

### F2 — `ProjectConfig` дублирует существующий конфиг-стек
В коде уже есть: `ArrayConfig`, `RangeConfig`, `WaveTimeConfig`, `SceneConfig`,
`SimulationConfig` + загрузка YAML через `YamlConfigSource`/`ConfigSource`
(`core/config/__init__.py`). Спека вводит `ProjectConfig` «единый источник правды»
**без сверки** с этим — риск «плодить сущности» (правило Alex).

**Фикс (внесён):** `ProjectConfig` **агрегирует** существующие
`ArrayConfig`/`RangeConfig`/`WaveTimeConfig`/`SceneConfig`, а **не заменяет** их;
YAML грузится существующим `YamlConfigSource`, новый лоадер не плодим.

### F3 — путь `jammers_rf` неточен + есть второй слой помех
Спека (§3, S3) пишет `core/generators/` `jammers_rf`. Реально —
`core/generators/waveforms/jammers_rf.py` (сигнал-уровень, 6 помех P5). Плюс
существует **отдельный** `core/generators/jammers.py` (куб-уровень:
`DrfmComb`/`BarrageJammer`/`HamEmitter`). Спека их не различает.

**Фикс (внесён):** уточнён путь + приписка, что куб-уровневый `jammers.py` — не трогаем
(берём как конвенцию параметров, ровно как решено в P5).

## 🟠 Внутренние противоречия спеки

### F4 — ⛔ СХЕМА FFT БЫЛА НЕВЕРНА (сверено с патентом, гл. 3 + 4-бис)
Исходная спека (и мои первые правки) приписывали **скользящее окно W=16/шаг8 ОБЕИМ
веткам** и **общий 3D-FFT**. Патент это опровергает — ветки обрабатываются **по-разному**:

- **ЛЧМ (гл. 3):** вход `nx×ny×N` (N=1024…10000). **ДВА раздельных FFT**: ① дальностный
  FFT **глобально по всей Z, один раз** («после дечирпа цель — тон на всю длину Z,
  посегментная оконная обработка по дальности **НЕ требуется**»); ② угловой FFT **nx×ny
  поячеечно** на каждом бине. **Окна НЕТ.** Выход плотный `nx×ny×N`.
- **АМ (гл. 4-бис):** **локальный 3D-FFT** по скользящему окну `nx×ny×D` (глубина D=**16**
  дефолт, до **256**), **переменный шаг 8/16/32/64**. Окно/шаг — **только здесь**.
- Общее у веток — **куб-примитив `nx×ny×L` + объёмный токенизатор** (OS-CFAR 3D →
  3D-признаки → токен), а **НЕ** FFT-этап. Плоская токенизация L=1 — частный случай.

**Никаких «1250 кубов = 10000/8»** — это был артефакт ошибочного окна на ЛЧМ.

**Фикс (внесён):** §1 таблица веток, §2 конвейер, §3 роли, §5 стадии, §7 S5 — переписаны
строго по патенту (решение Alex 2026-07-15: «всё как в патенте, без упрощений»;
N=параметр 1024…10000; АМ дефолт nx×ny×16 с ростом до 256). `scheme.svg` помечен
устаревшим (перерисовка отложена).

### F5 — «ветвление только во фронт-энде» ломается: домены после ветвей разные
Спека: ЛЧМ фронт = «дечирп → range-FFT» (1 ось), АМ фронт = «3D-FFT» (3 оси). После
этого домены **разные** (ЛЧМ: простр. время × range-freq; АМ: полностью
преобразован), значит «низ общий» неверен без доп. простр. FFT для ЛЧМ.

**Фикс (внесён, уточнён по патенту):** «низ общий» — это **НЕ** общий FFT, а общий
**куб-примитив + объёмный токенизатор** (гл. 4-бис). FFT-этап у веток **разный**: ЛЧМ =
2 раздельных FFT (дальностный глобальный + угловой nx×ny), АМ = локальный 3D-FFT по окну.
`Fft3DModel` (`fftn`) реюзается как **ядро АМ-фронтенда**, но **не** для ЛЧМ. Домены
сходятся на уровне **куба `nx×ny×L`**, а не FFT. (См. F4 — обновлено.)

### F6 — `reduce_kz`: имя третьей оси
Третья ось куба — **range** (частота быстрого времени), в коде `Axis("range")`
(`fft3d.py`). Спека зовёт её `kz`/`reduce_kz`. Не ошибка, но рассинхрон терминов.

**Фикс (внесён):** оставлено `reduce_kz` как редукция **по 3-й (range) оси** →
квадрат nx×ny (kx×ky, после pad — N_pad_x×N_pad_y) + `argmax` для уточнения дальности; добавлена сноска-синоним.
