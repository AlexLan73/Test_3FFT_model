## 0. Главная архитектурная находка анализа

**Web-панель = ТРЕТИЙ рендерер уже существующей истории тактов ex4.**

Ex4Flight уже сегодня: (а) считает конвейер по тактам и складывает ВСЁ в
`self._history: list[TactRecord]` (точки после null, истина, треки с историей,
срезы с 6 признаками, полоса/null); (б) из ОДНОЙ истории рендерит GIF×2 + last_frame
(правило R5 — «второй раз не считать»). Web-страница — это ещё один потребитель той же
истории: **сериализовать `TactRecord` → JSON → встроить в HTML** (механика v1).
Никакого нового конвейера/физики — только сериализация + JS-рендеры трёх элементов канона.

```
Ex4Flight._run_ticks (КАК ЕСТЬ) → history
        ├─ GIF trail / GIF clean      (есть)
        ├─ last_frame.png, trajectory (есть)
        └─ NEW: to_json(history) → web/index.html   (самодостаточный, офлайн)
```

## 1. Разбор канона §1.6 по элементам → маппинг на web

### ① 3D-сцена (слева): движение объектов и помех
Что в каноне (last_frame смотрела глазами): облако детекций такта (цвет=дБ, turbo),
истина цель/носитель (треугольники), **barrage = линия вдоль всей оси позиции** на своём
(kx, ky), оси kx × позиция × ky, ракурс (18°, −60°).

Web-реализация (plotly дома НЕТ → vanilla canvas):
- рёбра куба + 3 оси с подписями; проекция «поворот Rz(az)·Rx(el) → перспектива»;
- **вращение мышью (drag), зум колесом** — то, чего нет у matplotlib-GIF, ради этого web;
- детекции — точки цвета turbo по дБ; истина — треугольники; barrage — толстая
  полупрозрачная линия; хвост цели K тактов — затухающая ломаная в 3D;
- ~90 строк JS (матрица 3×3 + сортировка точек по глубине), без библиотек.

### ② Поле nx×ny (справа): плоское представление с хвостами 8 тактов
Канон: треки со следом K=8 (размер+альфа растут к голове), № у устойчивых (≥3 тактов),
✈ если `is_moving`; гребёнка — полый квадрат; barrage — рамка (дрейфует по тактам).
Web: canvas-скаттер 1:1 по `_compose_frame` (логика alpha=0.15+0.85·(j+1)/n — перенести
буквально); клик по треку = выбор (подсветка в 3D/срезах/таблице).

### ③ Ряд тепловых срезов + ТАБЛИЦА параметров (низ) — «по-взрослому»
Канон: на каждый трек — тепловое окно (глубина 16) с рамкой-кружком №, заголовок
`№ · окно(позиция) · kx · ky · ЛЕТИТ/—`, под ним 6 признаков §4.11.
**Alex: «тот кусочек с таблицей будем использовать»** — реализация = HTML-таблица v1
(строки-треки, моноширинные числа, подсветка выбранного), а не текст-в-заголовке PNG:

| № | окно | kx | ky | PR | Hoyer | MainFrac | LobeRatio | MaxMean | Energy | вердикт |
|---|------|----|----|----|-------|----------|-----------|---------|--------|---------|

- **«с описанием параметров»**: у каждого заголовка столбца — тултип (title) + блок-легенда
  (сворачиваемый) с одной строкой смысла на признак: PR — пик/фон; Hoyer — разреженность;
  MainFrac — доля главного лепестка; LobeRatio — боковики/главный; MaxMean — контраст;
  Energy — энергия окна; вердикт — «летит» ИЗ ТРЕКА (`is_moving`, §4-бис.4), не из куба.
- тепловые мини-карты срезов — canvas-heatmap (turbo, vmin −25..0), рамка вокруг
  углового бина трека; клик = выбор трека.

### ④ Контролы (из v1, работают уже сейчас)
play/pause · скорость 0.5/1/2× · слайдер тактов · трейл K вкл/выкл · выбор трека.
Строка статуса такта: `такт i/N · полоса@(kx,ky)→null` (канон заголовка 3D).

## 2. Данные: сериализация TactRecord → JSON (что и сколько)

| Поле такта | Источник (есть) | Объём/такт (64×64, 4 среза) |
|---|---|---|
| точки после null | `rec.points` (CoarsePoint: kx, ky, pos, db) | ~10-40 × 4 числа |
| истина | `rec.truth` {target, comb, barrage} | 9 чисел |
| треки | `rec.tracks` (id, kx, ky, history, is_moving) | ~10 × (4 + K·4) |
| полоса | `rec.banded`, `band_angle` | 3 |
| срезы | `rec.slices` (`energy_db` полный nx×ny!) | ⚠️ 4×4096 = 16k чисел |

⚠️ Единственная тяжесть — `energy_db` полного поля на срез: 30 тактов × 4 × 4096 ≈ 500k
чисел ≈ 3-4 МБ JSON. **Предложение: кроп окна ±8 бинов вокруг трека (17×17=289)** →
~35k чисел на всю историю (×14 легче), на мини-карте всё равно смотрят окрестность трека.
(В §7 — вопрос; вариант Б: полный, страница потолстеет до ~5 МБ, браузеру всё равно.)
