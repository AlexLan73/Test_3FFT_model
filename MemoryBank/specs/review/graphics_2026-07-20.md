## Находки (severity ↓)

### [MED] DRY (правило 06) — `core/graphics/angular_map.py:30-31` дублирует `core/graphics/layout.py:15-16`

**Что:** `AngularMapVisualizer.render()` хардкодит подписи осей прямо в коде:
```python
ax.set_xlabel("kx (азимут)")
ax.set_ylabel("ky (угол места)")
```
Это **дословно** те же строки, что уже заданы как единый источник истины в
`AxisLayout._LABELS`:
```python
_LABELS: dict[str, str] = {
    "kx": "kx (азимут)",
    "ky": "ky (угол места)",
    ...
}
```
**Почему нарушение:** правило `.claude/rules/06-graphics.md` прямо требует: «Человекочитаемые
подписи/лимиты осей — в `AxisLayout`». `angular_map.py` вообще не импортирует `AxisLayout` и не
использует `resolve()`, хотя работает с тем же `SpectralCube` (`cube.kx`, `cube.ky`) — источник
подписи раздвоен. Пока текст совпадает случайно; при следующей правке подписи в одном месте
(например, косметика "азимут, °") второй останется рассинхронизирован — тот же класс проблемы,
что описан в правиле 07 (прецедент 2026-07-19: два источника истины → расхождение).

**Как исправить:** `xlabel, _, _ = AxisLayout(axis_x="kx", axis_y="ky", ...).resolve(cube, "kx")`
(или добавить в `AxisLayout` метод `label_of(axis_key)`, не завязанный на `resolve()`'овые
лимиты) и использовать его в `AngularMapVisualizer`.

### [MED] ISP/Strategy-контракт заявлен, но не закреплён типом — `core/graphics/scene_points.py:57`

**Что:** Докстринг класса утверждает:
> «Контракт тот же (Strategy): `render(...) -> Figure`, на диск НЕ пишет»

но `class ScenePointsVisualizer:` **не наследует** `Visualizer(ABC)` (`core/graphics/visualizer.py:11`),
и сигнатура `render(self, points: list[ScenePoint], markers=(), title="", figsize=..., ax=None) -> Figure`
не совместима с абстрактным `render(self, cube: SpectralCube) -> Figure`. Все остальные
matplotlib-стратегии в кластере (`CubeScatterVisualizer`, `AngularMapVisualizer`,
`RangeProfileVisualizer`) честно наследуют `Visualizer`.

**Почему нарушение:** не сам факт другого входа (это оправдано — вход другого рода: список точек,
а не куб), а то, что докстринг называет это «тем же контрактом (Strategy)», создавая ложное
впечатление формального полиморфизма там, где на деле — duck typing без базового класса/протокола.
Ничто не мешает будущему коду попытаться положить `ScenePointsVisualizer` туда, где ожидается
`Visualizer` (по аннотации `list[Visualizer]`), и получить ошибку в рантайме, а не на этапе
тайпчека.

**Как исправить:** либо завести отдельный `Protocol`/ABC (`PointsVisualizer` с
`render(points, markers, ...) -> Figure`) и наследовать его явно, либо в докстринге чётко написать
«параллельная, НЕ совместимая с `Visualizer` иерархия» вместо «тот же контракт».

### [LOW] Скрытый флаг через служебный matplotlib-атрибут — `core/graphics/scene_points.py:91`

**Что:**
```python
if ax.get_gid() != "no-colorbar":
    fig.colorbar(sc, ax=ax, shrink=0.7, label="дБ")
```
Управление «рисовать ли colorbar» протащено через `Axes.set_gid()` — атрибут matplotlib,
предназначенный для SVG group-id при экспорте, а не для передачи прикладных флагов. Источник
флага — `demo/ex2_am_square/example.py:376` (`ax.set_gid("no-colorbar")`), т.е. вызывающий код
снаружи `core/` неявно управляет поведением рендера через магическую строку, не входящую в
сигнатуру `render()`.

**Почему нарушение:** нарушает явность публичного контракта Strategy — управляющий параметр не
виден в сигнатуре `render(...)`, а спрятан в побочном атрибуте переданного `ax`. Хрупко: опечатка
в строке `"no-colorbar"` не даст ошибки ни на одном конце.

**Как исправить:** явный параметр, например `render(..., show_colorbar: bool = True)`.
