# Ревью: models

Кластер: `core/models/**` (tokenizer, anti_barrage, tracking, targeting, angular_fft,
base, classification, fft3d, range_fft, result, windows).

## Находки (severity ↓)

### [HIGH] DIP/OCP — `AntiBarragePipeline` завязан на конкретный класс `SubspaceNuller`, нет общей абстракции для семейства "nuller"
**Файл:** `core/models/anti_barrage/pipeline.py:44` (сигнатура `__init__`), в связке с
`core/models/anti_barrage/nuller.py` (`SubspaceNuller`) и `core/models/anti_barrage/mvdr.py`
(`RobustMvdrNuller`).

**Что:** В пакете `anti_barrage` два независимых класса с ИДЕНТИЧНЫМ публичным контрактом
`apply(datacube: np.ndarray) -> np.ndarray` (тот же вход/выход, та же семантика "подавить
помеху") — `SubspaceNuller` (EVD-проекция) и `RobustMvdrNuller` (MVDR/Capon). Но общего
`ABC`/`Protocol` для этой роли нет (в отличие от `RadarModel`, `CubeClassifier`, `Arbiter`,
`Tracker`, `Targeting` в соседних модулях — везде есть базовый класс, здесь его пропустили).
`AntiBarragePipeline.__init__(self, nuller: SubspaceNuller, fft_model: RadarModel, cfar:
CaCfarDetector)` типизирует параметр КОНКРЕТНЫМ классом `SubspaceNuller`.

Подтверждено по кодовой базе: `RobustMvdrNuller` протестирован только изолированно
(`tests/test_mvdr.py`), но НИ РАЗУ не подключён к `AntiBarragePipeline` — grep по проекту
не находит совместного использования. Facade физически не даёт подставить вторую стратегию
подавления помехи, хотя она уже реализована и по духу — Strategy-альтернатива первой.

**Почему нарушение:** DIP — Facade верхнего уровня зависит от конкретики, а не от
абстракции. OCP — чтобы использовать `RobustMvdrNuller` в тракте, придётся МЕНЯТЬ
`pipeline.py` (тип параметра), а не просто передать другую реализацию через DI, как
задумано во всех остальных Strategy проекта (`RadarModel`, `CubeClassifier`, `Arbiter`).
Это ровно тот анти-паттерн, который правило `05-python-style.md` явно запрещает
("новая помеха = подкласс ... фабрика тракт не трогаем").

**Как исправить:** Выделить `class Nuller(ABC): def apply(self, datacube) -> np.ndarray`
(и, если нужно, `def report(...)` отдельным протоколом/опционально) в
`core/models/anti_barrage/`, сделать `SubspaceNuller`/`RobustMvdrNuller` его подклассами,
изменить сигнатуру `AntiBarragePipeline.__init__` на `nuller: Nuller`.

### [MED] DRY — дублирование вычисления ковариации (+diagonal loading) и reshape между `SubspaceNuller` и `RobustMvdrNuller`
**Файлы:** `core/models/anti_barrage/nuller.py:104-128` (`_reshape`, `_eigh`) и
`core/models/anti_barrage/mvdr.py:47-65` (`_reshape`, `_cov`).

**Что:** Метод `_reshape` (datacube (nx,ny,K) → x_mat (M,K), astype complex128) —
байт-в-байт одинаковый код в обоих файлах. Блок построения ковариации с diagonal loading
(`R = X@X.H/K`, затем `R + loading·(tr(R)/M)·I`) повторён почти дословно: `nuller.py:118-125`
внутри `_eigh` и `mvdr.py:57-64` внутри `_cov` — только у `SubspaceNuller` следом идёт
`eigh`, у `RobustMvdrNuller` — `solve`.

**Почему нарушение:** Правило проекта `.claude/rules/07-math-in-core.md` требует ОДИН
источник истины для математики именно чтобы избежать дублирования формул в двух местах —
здесь формула ковариации+loading продублирована буквально. Риск: правка формулы (например,
масштаба loading) в одном файле и забытая во втором — расхождение поведения между двумя
"вариантами одной и той же math", тот самый сценарий, о котором предупреждает правило.

**Как исправить:** Вынести `_reshape` и `_covariance_with_loading(datacube, loading) -> R`
в общий модуль (например `core/models/anti_barrage/covariance.py`, Pure Fabrication),
переиспользовать в обоих классах.

### [LOW] Несогласованный стиль импорта — абсолютный импорт вместо относительного
**Файл:** `core/models/anti_barrage/cfar.py:26` — `from core.models.result import SpectralCube`.

**Что:** Все остальные модули кластера (`tokenizer/tokenizer.py:15`, `targeting/cycle.py:19`,
`anti_barrage/pipeline.py` и т.д.) используют относительные импорты внутри `core.models`
(`from ..result import SpectralCube`, `from .beam import ...`). `cfar.py` — единственное
исключение, тянет через абсолютный путь `core.models.result`.

**Почему нарушение:** Не является багом сейчас (пакет `core` действительно на верхнем
уровне), но ломает единообразие модуля и делает код чувствительным к тому, как именно
запущен интерпретатор/наличию `core` в `sys.path` — остальной кластер от этого не зависит.

**Как исправить:** Заменить на `from ..result import SpectralCube` для консистентности.

### [LOW] Stringly-typed решения (`decision`, `reason`, `kind`) без enum
**Файлы:** `core/models/tokenizer/tokens.py:15-19` (`SliceLabel = str`, `RangeKind = str`),
`core/models/tokenizer/arbiter.py:50-51` (`TargetDecision.decision: str`, `.reason: str`).

**Что:** Ключевые дискриминаторы (`"target"/"false"/"jammer"`, `"target"/"comb"/"barrage"`,
десяток строковых `reason`) — сырые `str`, сравниваемые литералами в разных файлах
(`arbiter.py`, `tracker.py:91`, `beam.py:82`, `cycle.py`). Опечатка в литерале не поймается
статическим анализом.

**Почему нарушение:** Слабая типизация роняет ISP/безопасность контракта — потребитель
(`Tracker`, `Targeting`) обязан "знать" валидные строки без проверки компилятором/линтером.

**Смягчающее обстоятельство:** это осознанный выбор ради кросс-языковой сериализации
(msgpack/C++, см. докстринги `TargetDecision`/`Track`) — не менять сгоряча, но можно
добавить `Literal["target","false","jammer"]` в type hints (не меняет рантайм-поведение,
только статическая проверка) без потери кросс-языковой совместимости.

## Удачно

- **`OsCfarDetector`/`FeatureExtractor`/`RefinedPeak` (tokenizer)** — образцовые Value
  Object + Pure Fabrication: не мутируют вход, явные `dtype`, инвариантность признаков к
  размеру апертуры продумана и задокументирована со ссылкой на конкретный тест
  (`tests/test_tokenizer.py::test_pr_invariant_to_aperture_size`).
- **`CombinedArbiter` (arbiter.py:226-303)** — чистый Composite: комбинирует решения
  `EdgeArbiter`+`CodeArbiter` без дублирования их логики, явно документирует, какая
  геометрическая улика "сильная"/"слабая" — редкий случай, когда обоснование бизнес-правила
  и код не расходятся.
- **`SliceTriage`/`RuleBasedTriage` (triage.py)** — честно избежали нарушения LSP: вместо
  насильного наследования от `CubeClassifier` с другим контрактом завели параллельный узкий
  ABC той же Strategy-формы; девиация прямо описана в докстринге "для ревью Кодо".
