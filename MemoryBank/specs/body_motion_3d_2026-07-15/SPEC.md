## 1. Геометрия и оси

| Ось | Размер | Регулировка | FFT |
|-----|--------|-------------|-----|
| nx | 4…32 | не обязана быть квадратной (6×15) → zero-pad до 2ⁿ | kx (азимут) |
| ny | 4…32 | то же | ky (угол места) |
| fast-time (Z) | 1024…10000 | дальность | range |
| slow-time | n_pulses (тактов), дефолт 64 | импульсы CPI | Доплер |

**Ветки по модуляции — обе активны с старта. Схема FFT РАЗНАЯ (строго по патенту,
главы 3 и 4-бис, без упрощений):**

| Ветка | Вход | Обработка (фронтенд) | Выход |
|-------|------|----------------------|-------|
| **ЛЧМ** (точный) | 16×16×N, N=**1024…10000** | **2 раздельных FFT**: ① дальностный FFT **глобально по всей оси Z, один раз** (rocFFT, окна НЕТ — после дечирпа цель тон на всю Z); ② угловой FFT **16×16 поячеечно** на каждом бине дальности | 16×16×N (угол×угол×дальность, плотно) |
| **АМ** (грубый) | 16×16×N, N=1024…10000 | **локальный 3D-FFT** по скользящему окну **16×16×D** (глубина D=**16** дефолт, до **256**), **переменный шаг 8/16/32/64** (ручка грубо↔точно) | под-кубы 16×16×D (разреженно) |

**Формат входа обеих веток:** `16×16 плоскостей × N по дальности` — 16×16 это антенная
решётка (nx×ny → после углового FFT kx×ky), N — отсчёты быстрой (дечирпованной) оси,
N=1024…10000 (в патенте до ~1.3М; прототип — параметр 1024…10000).

- **ЛЧМ ≠ окно.** У ЛЧМ **нет** скользящего окна по дальности: дальность локализуется
  **одним глобальным** дальностным FFT по всей Z → N бинов, затем угловой FFT 16×16 на
  каждом бине. Выход **плотный** `16×16×N`.
- **Окно/переменный шаг — только у АМ.** АМ проходит объём **локальным 3D-FFT** по окну
  `16×16×D`. **Под-кубы идут внахлёст по половине окна** (дефолт: шаг = D/2 → overlap 50%;
  `D=16 → шаг 8`, `D=256 → шаг 128`). Переменный шаг **8/16/32/64** — ручка грубо↔точно:
  крупнее шаг = меньше нахлёст = грубее/дёшево, мелкий (8) — точно в помеченных зонах.
  Дефолт `D=16` (16×16×16), с возможностью увеличения до 256 (грубый скан `16×16×256`).
- Доплер (4-я ось, пачка `P` импульсов) заложен, FFT-этап — заглушка (torch-этап).

## 2. Конвейер: фронтенд-заполнение ветвится, куб+токенизатор общие

Ключ патента (глава 4-бис): **токен формируется на объёме** (куб «угол×угол×дальность»
`16×16×L`), а **способ заполнить объём — сменный интерфейс (фронтенд) под тип сигнала**.
Общее у веток — **куб-примитив и объёмный токенизатор**, а НЕ FFT-этап (он разный).

```
ЛЧМ 16×16×N ─┐  фронтенд-заполнение (Strategy, РАЗНОЕ на ветку):
             │    ЛЧМ (LfmToCube): ① дальностный FFT глобально по Z (1 раз, без окна)
             │                     ② угловой FFT 16×16 поячеечно на каждом бине
             │                     → куб 16×16×N  (плотно, точно)
АМ  16×16×N ─┤    АМ (AmToCube):   локальный 3D-FFT по окну 16×16×D (D=16…256),
             │                     нахлёст ½ (шаг = D/2, дефолт D16→шаг8; переменный 8/16/32/64)
             │                     → под-кубы 16×16×D  (грубо, разреженно)
             ▼
   объёмный примитив «угол×угол×дальность» 16×16×L   ← общий, безразличен к фронтенду
             ▼  объёмный токенизатор (ОБЩИЙ): OS-CFAR 3D → 3D-признаки → токен
   токены (k_x,k_y,k_z + позиция окна + признаки + метка)
             ▼
   детектор / визуал / панель   ← ОБЩИЙ аппарат для обеих веток
```

Ветвится **только заполнение куба** (`LfmToCube` = 2 FFT / `AmToCube` = локальный 3D-FFT
по окну). Дальше — **один** объёмный токенизатор над кубом `16×16×L` (плоская
токенизация L=1 — частный случай). `k_z` — 3-я (дальностная) ось куба.

> 📌 Прототип P6: сначала можно свести «низ» к упрощённой визуализации (reduce по
> range-оси → квадрат 16×16 энергии + argmax дальности) как контрольный вид; полный
> объёмный токенизатор (OS-CFAR 3D + 3D-признаки, глава 4-бис) — на этапе детектора.

## 4. Единая конфигурация и шина данных (ключевое)

**`ProjectConfig`** — один источник правды для всего проекта: геометрия, окно/шаг,
n_pulses, помехи, транспорт, визуал. **Агрегирует существующие VO** (`ArrayConfig`,
`RangeConfig`, `WaveTimeConfig`, `SceneConfig`) — не заменяет их (F2). Грузится из YAML
**существующим** `YamlConfigSource`/`ConfigSource` (`core/config/`), новый лоадер не
плодим; загрузку прокидываем **через `DataContext`**.

**`DataContext` — расширяем существующий Facade, не переписываем (F1).**
Сейчас в коде это `save_cube`/`load_cube` над `CubeRepository` — эту сигнатуру
**сохраняем** (обратная совместимость). Персистентность (Repository) и рантайм-шина
(pub/sub) — **разные ответственности** (SRP), поэтому шину выносим в отдельный
`Blackboard`/`MessageBus` (Subject), а `DataContext` его **композирует**.

Единая централизованная точка обмена: **любое** чтение/запись/раздача данных идёт **к
нему или через него** — файлы (load/save через Repository), конфиг, состояние сцены,
поток от продюсера к наблюдателям (через шину). Нет прямого I/O в обход `DataContext`.

- загрузка/сохранение (Facade + Repository): конфиг, кубы, кадры, датасеты;
- держит текущее состояние: конфиг, куб, квадраты, кадры, треки целей;
- `bus.publish(key, data)` — кладёт данные в шину и **уведомляет наблюдателей** (Subject);
- `bus.subscribe(observer)` — регистрация наблюдателя.

**Правило:** весь визуальный код — **Observer**. Ничего не рисуется само по себе;
рендер срабатывает **после помещения данных в шину**. Продюсеры (сцена/GPU) и
наблюдатели (панель, plotly, matplotlib) не знают друг о друге — только шину.

> 🖼️ Схема конвейера и рантайма — `scheme.svg` (в этой папке). ⚠️ **Устарела** — нарисована
> под ошибочную схему «окно для обеих веток»; перерисовать под патентную (ЛЧМ = 2 FFT без
> окна, АМ = локальный 3D-FFT по окну 16×16×D). Отложено (решение Alex 2026-07-15).

```
producer (SceneServer/GPU) ──publish──▶ DataContext (шина) ──notify──▶ Observer (Visualizer/панель)
                                              ▲ ProjectConfig (YAML)
```

## 5. Квадраты и разворачиваемая закладка

> 🧭 **Конвенция 3D-визуала (везде):** дальность (range) — по **горизонтали** (пол сцены,
> уходит вдаль); kx (азимут) — вбок; ky (угол места) — вверх. Динамика: `demo_body_motion.py`
> пишет GIF, флаг `--live` — живое окно, `.html` (plotly) — вращение мышью.
>
> 🟢 **Боевой фронт живой системы — ДВА, движок один (решение Alex 2026-07-15):** веб-дашборд
> (браузер, Three.js/HTML) + десктоп (Dear PyGui), оба — Observer'ы над `Transport` (ZMQ+WS,
> fan-out). Продюсер публикует один раз. Детали — P6.
>
> 🟢 **Штатный вид данных (решение Alex 2026-07-15): сырые кубы НЕ визуализируем.**
> Показываем **тепловую карту 16×16 после углового 2D-FFT** — `|FFT2D(апертура)|²` с
> `fftshift` (цель = точка в `(kx,ky)`, а не «стена»). **Окно Хэмминга по апертуре
> ОБЯЗАТЕЛЬНО (обе оси — горизонт/вертикаль, решение Alex)** — подавляет sinc-лепестки,
> точка чистая. Это и есть «квадрат» (§5, гл.3 п.2 патента). Блок «окрестность ±N» (§5.1)
> — реализован-прототип: 5 окон (K−2…K+2) с номером бина + расч. дальностью + квадрат с
> токенами; расч. дальность из сырого бина — приближение, точную даёт дечирп+range-FFT (P5).

**Стадии (токены = метки; сейчас из ground-truth, позже — выход детектора):**

- ЛЧМ: `Q1` после дальностного FFT (профиль по Z) · `Q2` угловые плоскости 16×16 после
  углового FFT (на выбранных бинах дальности) · `Q3` квадрат с токенами.
- АМ: `Q1` сырьё после АЦП (окно `16×16×D` пунктиром) · `Q2` плоскости под-куба после
  локального 3D-FFT · `Q3` квадрат с токенами.

**Закладка «окрестность объекта» (±N плоскостей, дефолт N=5):** объект в плоскости K →
показываем K−2…K+2. Блок на каждый сигнал (кол-во блоков = кол-ву сигналов), 3 ряда:
теплокарта (2D/3D-FFT) → место в квадрате → описание точек (токены + подписи).
Заград — отдельный блок с пометкой углов.

## 6. Работа программы: сегодня Windows / завтра Linux

> 🎯 **Проектируем сразу под C++ (кросс-язычно py↔C++), с прицелом на дальнейшее развитие
> (решение Alex 2026-07-15).** Python-прототип — референс, но **контракт обмена язык-
> нейтрален с S1**: схемы сообщений MessagePack и модель данных панели (`Field`/`Cell`/
> `Element`) — единый контракт, чтобы C++/GPU-продюсер и панель встали **без изменения
> протокола/шины/конфига**. Не завязываемся на python-специфичный pickle/формат; ZMQ+
> MessagePack выбраны именно ради кросс-язычности. Именование/структуры — портируемые.

**Сегодня (Windows, дом) — чистый python:**

- backend `numpy_backend`; генерация + движение + FFT на CPU.
- продюсер и панель — оба python на localhost, транспорт ZMQ+MessagePack.
- панель — Dear PyGui (live-квадраты), plotly — 3D-обзор. Без torch/ROCm.

**Завтра (Linux, работа) — GPU:**

- backend `hip_backend` + rocFFT batch (N кубов за вызов); `.so` под **cp313**
  (numpy-путь генерации/rocFFT, без torch). ⚠️ torch-среда (классификатор/Доплер) —
  **cp312** (колёса torch-ROCm cp312, правило проекта). Два разных venv, не путать (F7).
- продюсер может быть C++/GPU и **публиковать в тот же ZMQ+MessagePack** — панель
  (Observer) не меняется. Переключение backend — через factory, код сцены не трогаем.
- torch-среда (cp312) — только для обучаемого классификатора/Доплера.

Панель, шина `DataContext`, `ProjectConfig`, протокол — **одни и те же** на обеих
платформах; меняется только вычислительный backend и источник данных.

```
[Windows] numpy-продюсер ─ZMQ+msgpack─▶ DataContext ─notify─▶ Dear PyGui + plotly
[Linux]   C++/GPU (rocFFT) ─ZMQ+msgpack─▶ DataContext ─notify─▶ Dear PyGui + plotly
                                              ▲ ProjectConfig (общий YAML)
```
