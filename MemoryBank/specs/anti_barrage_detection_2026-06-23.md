# Спека: распознавание летящей цели в заградительном шуме (barrage)

> Дата: 2026-06-23 · Автор: Кодо · Привязка: проект **radar3d**
> (3D-БПФ куб угол-угол-дальность на матричных GPU + каскад RuleBased→3D-CNN).
> Задача: отделить отклик цели от заградительной шумовой помехи (`barrage`),
> где детерминированный `RuleBasedClassifier` путает barrage с `ham` (заливка).

---

## 0. Краткий вывод (TL;DR)

**Ключ к barrage — асимметрия локализации.** Заградительная помеха
**локализована по УГЛУ** (один источник/направление), но **размазана по
ДАЛЬНОСТИ/Доплеру**. Цель — локализована **и по углу, И по дальности** (а если
есть Доплер — имеет ненулевую скорость). Именно эта разница и эксплуатируется
всеми методами — и классическими, и нейросетевыми.

Для нашего куба (две угловые оси `kx,ky` 16×16 = 256 простр. каналов + дальность)
**самый сильный рычаг — пространственное (угловое) подавление**: поставить
**ноль ДН в направлении помехи** (eigen-subspace / oblique projection / MVDR).
Цель под другим углом — выживает. Это ровно то, что наш куб позволяет делать
«бесплатно»: угловые оси уже есть. 3D-CNN — второй каскад, учит сигнатуру
угол-угол-дальность и берёт случаи, где геометрия не разделяет (mainlobe-помеха).

## 1. Физика задачи и привязка к нашему кубу

| Признак | Цель (`target`) | Заградительная (`barrage`) |
|---------|-----------------|----------------------------|
| Угол (kx,ky) | локализована (1 бин) | локализована (1 направление) |
| Дальность | **локализована** (пик) | **залита вся ось** |
| Доплер/скорость | ненулевая (движется) | широкий/некогерентный шум |
| Ковариация по решётке | ранг сигнала низкий | сильное собственное значение под углом помехи |

Наш `BarrageJammer` (core/generators/jammers.py) генерит ровно это: когерентный
по элементам шум, наведённый на `(kx,ky)`, залитый по всей дальности →
в кубе это «столб» энергии по дальности под фиксированным углом. Подтверждено
численно (спека energy_zones: barrage ≈ 17/15/14% по зонам дальности, `бин0≈0дБ`).

> ⚠️ Важный нюанс нашей модели: **Доплера пока нет** — один снимок CPI
> (fast-time дерамп), без слоу-тайм/последовательности импульсов. Значит
> классический STAP/MTI (угол-Доплер) напрямую **не применим**, пока не добавим
> ось импульсов. Зато простр. (угловое) подавление применимо в полный рост:
> у нас 256 элементов решётки → богатая угловая ковариация.

## 2. Классические DSP-алгоритмы (проверяемые, без обучения)

### 2.1. Пространственное адаптивное подавление — eigen-subspace / oblique projection ⭐
Самый релевантный нам класс. Идея (источник: Sensors PMC9654238, IET 2024 Han):
- Ковариация по решётке **R = U_s Λ_s U_sᴴ + U_n Λ_n U_nᴴ**: большие собств.
  значения = сигнал+помеха, малые = шум.
- Помеха (мощный «столб») даёт **доминирующее собственное значение** →
  её собственный вектор `u_J` ≈ steering-вектор направления помехи.
- Строим **ортогональный проектор** `P⊥ = I − u_J(u_Jᴴu_J)⁻¹u_Jᴴ` и применяем
  к данным: `y = P⊥ x` — помеха зануляется.
- **Oblique projection** (косая) сохраняет цель без искажения:
  `B = U_Sa(U_Saᴴ P⊥ U_Sa)⁻¹ U_Saᴴ P⊥`, где `B·a(θ_цель)=a(θ_цель)`, `B·u_J=0`.
- Для **mainlobe-помехи** (помеха близко к цели по углу) — реконструкция
  ковариации + MUSIC для точного DOA + null-constraints (иначе адаптивная ДН
  «ломается»: искажение главного луча).
- **Предел:** разделяет 2–3 помехи; деградирует при неточном DOA и при
  близких по углу/мощности помехах.

➡️ **Нам:** считаем ковариацию `R (256×256)` по кубу, ищем доминирующее с.з.,
проецируем `P⊥` по угловым осям — barrage-направление гаснет, цель остаётся.
Дёшево на матричных ядрах (EVD/SVD — это GPU-friendly BLAS).

### 2.2. Sidelobe canceller (SLC) — классика ECCM (патенты)
Главная антенна + вспомогательные каналы; адаптивный фильтр вычитает помеху,
пришедшую по боковым лепесткам. Патенты: **US5351058A** (general-purpose SLC),
**US4586048A** (IF SLC против низкоскважного импульсного), **US3982245**
(коррелятор с весами), **EP1159635B1** (nulling в blanker+main beam),
**US5343211A** (фазовая решётка с «широким нулём»). Для barrage в боковых
лепестках — рабочая лошадка; для mainlobe — недостаточно (нужен §2.1).

### 2.3. STAP (space-time adaptive processing) — если добавим Доплер
Совместная фильтрация угол×Доплер. Barrage «localized in angle, spread over all
Doppler» → STAP ставит угловой ноль, сохраняя медленную цель. Нужна оценка
ковариации клаттер+шум (≥2×DOF выборок — дорого, неоднородность вредит).
Алгоритмы: SMI, DPCA/ADPCA, sparse-Bayesian STAP (PMC9332471), random-matrix STAP
при малой выборке (arXiv 2202.04878). **Требует слоу-тайм** — для нас это extension.

### 2.4. MTI/MTD + CFAR — детектор после подавления
После пространственного подавления — обычный тракт: пик по дальности → CFAR
(адаптивный порог по шуму) для решения «цель/нет». MTI/MTD выделяют **движущуюся**
цель доплеровской фильтрацией (опять же нужен импульсный пакет). PRF/PRI-аджилити
ломает когерентные помехи.

## 3. Глубокое обучение (для случаев, где геометрия не разделяет)

### 3.1. RD/cube-денойзер barrage ⭐ (прямой аналог нашего 3D-CNN)
**IET RSN 2024, Wu — «lightweight denoising network для multiple barrage jamming
в range-Doppler domain»** — самый близкий к нам результат:
- Вход: «грязная» RD-картина (цель + barrage). Архитектура: **encoder-decoder**
  с **depthwise + pointwise свёртками** (легковесная, multi-resolution).
- Учится **подавлять помеху и усиливать эхо цели**; давит 4 типа одиночного
  barrage + 6 типов составной помехи.
- Наш аналог: тот же encoder-decoder, но на **3D-кубе угол-угол-дальность**
  (Conv3d) — денойзинг/сегментация «где цель».

### 3.2. CNN-распознавание типа помехи (классификатор каскада)
Вход — **время-частотные образы** помехи; точность 99%+ даже при JNR −3 дБ:
- **RDJCNN** (микро-CNN, ESWA 2023) — классификация активных помех, лёгкая.
- **Hybrid 3D-CNN-Transformer** (IET RSN 2024) — быстрое распознавание, экономит
  ресурсы (важно для GPU-каскада).
- **Open-set** распознавание (Preprints 2025) — ловит **неизвестные** типы помех
  (для нас: «это не из 5 классов»). 
- **AWSPNet** (arXiv 2510.18422) — wavelet-scattering + прототипная сеть,
  одновременно распознаёт цель MIMO и давит помеху.

### 3.3. Прочее
- **VAE** для compound-jamming (Remote Sensing 2024) — представление, ловит
  помехи, которые «не берутся правилами» (наш случай barrage↔ham!).
- **Temporal-CNN + attention** — слабая цель в клаттере без роста ложных тревог.
- **Deep-RL ECCM** — адаптивный выбор зондирующего сигнала (стратегический слой).
- **PSF-prior сети** — обобщение на невиданные настройки джаммера (робастность).

## 5. 🎯 Как это помогает radar3d — конкретный план

### Каскад (расширение текущего RuleBased→3D-CNN):

```
куб (kx,ky,range)
  │
  ├─[A] оценка ковариации R(256×256) по угловым каналам
  │      → доминирующее с.з. = barrage? (порог по λ_max/λ_шум)
  │
  ├─[B] eigen/oblique projection P⊥: занулить угол помехи  ← §2.1
  │      → «очищенный» куб (цель под др. углом выживает)
  │
  ├─[C] CFAR-детектор по дальности в выживших угловых ячейках  ← §2.4
  │      → есть локализованный пик ≠ заливка → ЦЕЛЬ
  │
  └─[D] 3D-CNN (наш Cnn3DClassifier) — финальный арбитр там,
         где [A–C] не уверены (mainlobe-помеха, low-SNR)  ← §3.1/3.2
```

### Что внедрить по приоритету (наш порядок: работает→корректно→быстро):

1. **`SubspaceNuller`** (новый источник-фильтр, §2.1) — Pure Fabrication в
   `core/models/`: на входе сырой куб, считает `R`, проецирует `P⊥`.
   Детерминирован, доказуем, **разделит barrage и target по геометрии** там,
   где RuleBased пасует. Малый объём кода, большой эффект. ⭐ Начать здесь.
2. **Метрика «локализация по дальности»** — уже есть (energy_zones из
   edge_scenes_demo). Формализовать как признак классификатора:
   заливка(occupancy высокая)+1 угол ⇒ barrage; пик+1 угол ⇒ target.
   → улучшит сам RuleBased **без сети**.
3. **3D-CNN денойзер/классификатор** (§3.1) — Conv3d encoder-decoder,
   обучить на наших сценах (target/barrage/comb/ham/empty). Берёт mainlobe-кейсы.
4. **(Extension) добавить слоу-тайм** (пакет импульсов) → откроет Доплер →
   STAP/MTI → разделение по скорости (летящий объект ≠ статичный шум).

### Почему это «ложится» на наш проект:
- Угловые оси куба = готовая основа для пространственного нуллинга (не надо
  ничего достраивать в физике).
- EVD/проекции и Conv3d — **матричные операции** → прямо на целевые GPU-ядра.
- Не ломает архитектуру: `SubspaceNuller` — ещё одна `Strategy`/Template-шаг
  перед классификатором; 3D-CNN уже предусмотрен (LSP с RuleBased).

## 6. Рекомендация и альтернативы

| Подход | Плюсы | Минусы | Вердикт |
|--------|-------|--------|---------|
| **Eigen/oblique projection** (§2.1) | доказуем, дёшев, GPU-friendly, разделяет по углу | mainlobe-помеха, нужен DOA | ⭐ **внедрять 1-м** |
| Признак локализации дальности (§2/energy_zones) | почти бесплатно, чинит RuleBased | не берёт mainlobe | ⭐ **внедрять параллельно** |
| 3D-CNN денойзер (§3.1) | берёт сложные/mainlobe, robust | нужен torch+данные+обучение | внедрять 2-м (каскад) |
| STAP/MTI (§2.3) | классика, по скорости | **нужен Доплер** (слоу-тайм) | extension |
| SLC (патенты §2.2) | проверено десятилетиями | sidelobe-only, аппаратно-ориентир. | референс идей |

**Итог:** связка **[пространственный нуллинг по собств.подпространству]** +
**[признак локализации по дальности]** даёт детерминированное разделение
target↔barrage уже сейчас; **3D-CNN** — второй каскад для mainlobe/low-SNR;
**Доплер (слоу-тайм)** — стратегический extension для «летящего» (по скорости).

## 7. Энергетическая дальнометрия постановщика — критический разбор

> Контекст: обсуждали тезис коллеги — «ИИ оценивает дальность до постановщика
> заградительной помехи по яркости шумового столба, как астроном по светимости
> звезды (стандартная свеча)». Разбор: что верно, что переоценено.

### 7.1. Что физически корректно
- Помеха идёт **в один конец** (постановщик → приёмник) ⇒ мощность падает как
  $1/R^2$ (а не $1/R^4$, как эхо цели). Уравнение постановщика:
  $$P_{rx}=\frac{ERP\cdot G_{rx}\cdot\lambda^2}{(4\pi)^2R^2}
  \;\Rightarrow\; R=\sqrt{\tfrac{ERP\cdot G_{rx}\cdot\lambda^2}{P_{rx}(4\pi)^2}}.$$
- Аналогия «стандартной свечи» точна: **без абсолютной светимости (ERP)
  дальность недостижима** из одного замера.
- Это известный класс ECCM — **Home-On-Jam (HOJ)**: self-screening jammer выдаёт
  своё **направление (AOA)** бесплатно; «навестись на источник помех» — штатно.

### 7.2. Что переоценено / ломается
- **ERP-по-сигнатуре — это ПРИОР, не измерение.** Ошибка лезет под корень:
  $$\frac{\delta R}{R}=\tfrac12\frac{\delta ERP}{ERP}.$$
  ERP ошиблись ×2 → дальность ±41%; ×10 (дрон vs контейнер РЭБ) → дальность ×3.2.
  Оценка «в разы», не метры.
- **Внутреннее противоречие (ключевой контраргумент Alex):** метод одновременно
  требует «шум равномерный, одинаковая сигнатура» (шаг 1) **и** «по сигнатуре
  угадываем тип → ERP» (шаг 2). Если сигнатура **одинаковая — угадывать ERP не по
  чему**. Классификатор-«свеча» работает только при **различных** спектральных
  отпечатках постановщиков. Чистый равномерный barrage намеренно лишает и времени
  прихода, и отпечатка ⇒ энергодальнометрия вырождается в угадайку по приору.
- **Одиночный куб ⇒ дальность до постановщика принципиально НЕнаблюдаема:**
  связка $ERP/R$ неразделима из одного замера мощности. Это **замысел** помехи —
  она отрицает дальнометрию. Строгий выход один — **движение во времени**.
- **Кинематика/RNN — самая сильная часть** (пассивная дальнометрия, bearings-only
  TMA): из серии сканов
  $$\frac{d\ln P}{dt}=-2\frac{\dot R}{R},\qquad \dot\theta=\frac{V_\perp}{R}$$
  можно вытащить $R$ **без знания ERP**, но нужна наблюдаемость (маневр/несколько
  сканов), сходится медленно.
- **Терминологическая путаница:** дальность **до постановщика** = дальность **до
  цели** только если помеха стоит **на цели** (self-protection). Для standoff —
  это разные дальности.

### 7.3. Иерархия надёжности (для нас)
| Что | Надёжность |
|-----|-----------|
| Угол на помеху (AOA) | ✅ бесплатно, точно |
| JNR/яркость ($\lambda_1/\sigma^2$, energy_zones) | ✅ меряем |
| Дальность по энергии+ERP-приор | ⚠️ грубо, ломается на одинаковом шуме |
| Дальность по кинематике (TMA/Доплер) | 🔧 строго, нужно время+движение |

**Вывод:** честный путь к дальности «летящего» — **не яркость столба, а Доплер +
многосканная кинематика**. Смыкается с вариантом «Доплер» из дорожной карты.

## 8. Расширение модели для симуляции этих идей (план B)

Чтобы вообще проверить энергодальнометрию/кинематику, в нашей модели **нечего
анализировать**: `BarrageJammer` = чистый белый шум + 1 угол + мощность, **без
сигнатуры**, и **нет оси времени/движения**. Что добавить (по слоям, без слома
архитектуры):

### 8.1. Сигнатуры постановщиков (чтобы «свеча» имела что классифицировать)
- Новый спец `BarrageSpec` → расширить: `bandwidth` (ограниченная полоса шума),
  `rotor_harmonics` (микродоплер винтов дрона), `erp_class` (тип носителя).
- В `BarrageJammer.contribute`: вместо белого шума — **полосовой** (фильтр по
  быстрому времени) + добавка узких гармоник. Тогда спектр столба несёт «отпечаток».
- Реестр ERP-классов (Value Object): `{drone: ~5Вт, pod: ~1000Вт, ...}` — приор.

### 8.2. Ось времени (последовательность кубов)
- `Synthesizer.build_sequence(scene, n_scans, dt)` → стек кубов $T_1..T_K$.
- Источники получают **кинематику**: `position(t)`, `velocity` → угол $(k_x,k_y)$ и
  яркость $P(t)\propto 1/R(t)^2$ меняются от скана к скану.
- Новый VO `Track` (углы+яркость по времени) — вход для TMA/RNN.

### 8.3. Доплер (слоу-тайм внутри одного скана) — см. §2.3/math §5
- `RangeConfig` → `+n_pulses`, `+pri`; `PointTarget._tone` → фаза $e^{j2\pi f_d T_r l}$
  с $f_d=2v_r/\lambda$; БПФ по импульсам → 4-я ось куба.

### 8.4. Оценщики (детерминир., проверяемые через TestRunner)
- `EnergyRanger` — $R=\sqrt{ERP/P_{rx}\cdot\text{const}}$ по приору ERP-класса
  (демонстрирует и **ограничения** §7.2 — намеренно покажем разброс).
- `KinematicRanger` (TMA) — из `Track`: $\dot\theta,\;d\ln P/dt$ → $R$. Строгий путь.
- Сверка: какой ранжер при каких условиях наблюдаем (маневр/JNR/число сканов).

### 8.5. Порядок (если решим делать)
1. §8.3 Доплер (он и так в дорожной карте, даёт скорость «летящего»).
2. §8.2 ось времени + `Track` (минимум для кинематики).
3. §8.4 `KinematicRanger` — честная дальность без ERP.
4. §8.1 сигнатуры + `EnergyRanger` — чтобы показать, где «свеча» работает, а где нет.

> ⚠️ Это заметное расширение физики модели — **только по согласованию** (правило
> 02-workflow: архитектуру/физику не менять без OK). Здесь — план, не реализация.
